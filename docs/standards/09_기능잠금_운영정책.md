# 09_기능잠금_운영정책

작성 기준: 2026-03-11  
적용 단계: 공식 5단계 완료 기준 / 8-0 준비정리 완료 기준

---

## 1. 문서 목적

이 문서는 OkeyGolf OS의 멤버십별 기능 잠금, 기능 노출, 연결 준비 상태, 운영자 override 원칙을 정의하기 위한 기준 문서다.

이 문서의 목적은 다음과 같다.

- trial, standard, pro, enterprise 등 멤버십별 기능 접근 정책을 명확히 한다.
- 준비중 기능을 전부 열어두지 않고, 실제 연결 상태에 따라 활성 / 잠금 / 숨김을 구분한다.
- 고객이 기능 상태를 오해하지 않도록 문구와 상태 표시 원칙을 고정한다.
- 외부 연동 기능(OpenAI, 아임웹, 알림/발송, 외부 수집 등)이 붙어도 기존 잠금 구조가 흔들리지 않게 한다.
- 운영자 전용 override와 고객용 기능 노출 정책을 분리한다.

---

## 2. 기능 잠금 운영 원칙

기능 잠금은 단순한 임시 장식이 아니라 실제 구독형 SaaS 운영 구조의 일부다.

원칙은 다음과 같다.

- 준비중 기능을 전부 해제하지 않는다.
- 실제 연결된 기능만 활성화한다.
- 연결 전 단계 기능은 상태를 구분해서 표시한다.
- 잠금과 숨김은 다르게 다룬다.
- 고객이 눌렀을 때 왜 안 되는지 이해할 수 있어야 한다.
- 운영자 override가 있더라도 기본 정책은 멤버십 기준을 따른다.
- 고객용 화면과 운영자용 화면의 통제 수준을 분리한다.

---

## 3. 잠금 상태 정의

기능 상태는 최소 아래 세 가지로 구분한다.

### 3-1. enabled

- 의미: 활성
- 설명: 정상 사용 가능
- 기준: 멤버십 조건, 연결 상태, 운영자 설정 조건이 충족된 상태

### 3-2. locked

- 의미: 잠금
- 설명: UI는 보이지만 현재는 사용할 수 없음
- 기준: 멤버십 부족, 권한 부족, 업그레이드 필요, 운영자 설정 미완료 등
- UX 원칙:
  - 이유를 짧게 보여준다
  - 필요 시 업그레이드 안내를 함께 보여준다
  - 실제 사용 가능한 것처럼 보이지 않게 한다

### 3-3. hidden

- 의미: 숨김
- 설명: UI에서 완전히 제거
- 기준: 현재 단계에서 고객에게 노출할 필요가 없거나, 운영자 전용 기능이거나, 구조상 아직 보여주면 오해가 큰 기능
- UX 원칙:
  - 고객에게 혼란을 줄 기능은 숨김이 우선이다
  - 운영자 전용 기능은 고객용 화면에서 숨김 처리한다

---

## 4. 멤버십 등급 운영 기준

현재 멤버십 등급은 아래를 기준으로 운영한다.

- `trial`
- `standard`
- `pro`
- `enterprise`

운영 원칙은 다음과 같다.

- 등급이 올라갈수록 기능 범위가 넓어진다.
- 전담 컨설턴트, 전달, 고급 시장조사, 외부 연동 후 실제 자동화 기능 등은 상위 등급 우선 구조를 따른다.
- 엔터프라이즈는 pro 기능을 포함하면서 멀티브랜치, 고급 권한, 조직 단위 확장 구조를 추가로 가진다.

## 5. 멤버십별 기본 정책

### 5-1. Trial

기본 방향:
- 체험 중심
- 핵심 구조를 이해하게 하되, 상용 핵심 기능은 잠금 유지

기본 활성 예시:
- 대시보드 조회
- AI 비서 일부
- 체크리스트/요약형 기본 결과
- 저장된 결과 기본 저장
- 현재 상태 확인 기능

기본 잠금 예시:
- 기본 생성 기능 상당수
- 결과 복사
- 결과 재생성
- 결과 내보내기
- 결과 공유
- 고급 시장조사
- 전담 컨설턴트 기능
- 결과 전달 기능

기본 숨김 예시:
- 운영자 전용 기능
- 엔터프라이즈 전용 기능
- 고급 외부 연동 관리

### 5-2. Standard

기본 방향:
- 일반 운영자가 실제 업무에 쓰기 시작할 수 있는 기본 상용 등급

기본 활성 예시:
- 기본 생성 기능
- 결과 저장
- 결과 복사
- 결과 재생성
- 결과 내보내기
- 결과 공유
- 기본 시장조사 기능

기본 잠금 예시:
- 전담 컨설턴트 일부 또는 전체
- 결과 전달
- 고급 시장조사
- 외부 연동 기반 고급 자동화 기능

기본 숨김 예시:
- 운영자 전용 관리
- 엔터프라이즈 전용 기능

### 5-3. Pro

기본 방향:
- 실무 사용을 넘어 실제 운영/전달/전담 요청까지 폭넓게 쓰는 상위 등급

기본 활성 예시:
- 전체 생성 기능
- 저장 / 복사 / 재생성 / 내보내기 / 공유 / 전달
- 전담 컨설턴트 요청
- 고급 시장조사
- 외부 연동 후 실제 연결 기능 일부

기본 잠금 예시:
- 멀티브랜치
- 고급 조직 권한
- 일부 엔터프라이즈 전용 기능

기본 숨김 예시:
- 운영자 전용 내부 관리

### 5-4. Enterprise

기본 방향:
- pro 기능 전체 포함 + 조직 단위 운영 확장

기본 활성 예시:
- pro 기능 전체
- 멀티브랜치
- 고급 권한
- 조직 단위 설정
- 향후 엔터프라이즈 전용 리포트
- 조직별 템플릿/지침 차등 적용

기본 잠금 예시:
- 없음 또는 계약형 기능만 별도

기본 숨김 예시:
- 운영자 내부 처리 기능은 고객용 화면에서 계속 숨김

## 7. 시장조사 관련 기능 잠금 정책

시장조사는 독립 메뉴로 유지하되, 깊이와 자동화 수준에 따라 잠금이 달라질 수 있다.

### 7-1. 기본 시장조사

- trial: 제한적 enabled
- standard: enabled
- pro: enabled
- enterprise: enabled

### 7-2. 고급 시장조사

예시:
- 외부 수집 기반 조사
- 대량 수집
- 고급 정리/비교 기능
- 고도화된 조사 결과 재가공

기본 방향:
- trial: locked
- standard: locked 또는 제한적
- pro: enabled
- enterprise: enabled

### 7-3. 외부 수집 기반 시장조사

현재 단계 기준:
- placeholder
- 실제 외부 연동 전에는 enabled로 보이면 안 된다

상태 표기 원칙:
- 연결 준비
- 미연결
- 운영자 설정 필요

## 8. 전담 컨설턴트 관련 잠금 정책

전담 컨설턴트는 프리미엄 서비스 성격이 강하므로 상위 등급 우선 구조를 따른다.

기본 방향:
- trial: locked
- standard: locked 또는 제한적 노출
- pro: enabled
- enterprise: enabled

원칙:
- 고객이 기능 자체를 모르는 것보다, 잠금 상태와 필요 조건을 이해하는 것이 더 중요한 경우 locked로 노출한다.
- 단, 너무 이른 단계에 과도한 노출로 혼란이 생기면 일부는 hidden도 가능하다.

## 9. 외부 연동 관련 잠금 정책

외부 연동은 실제 연결 상태와 멤버십 조건을 함께 본다.

### 9-1. OpenAI 기반 실제 생성

현재 단계 기준:
- 구조 준비
- 실제 연결 전에는 placeholder 또는 연결 준비 상태

연결 후 기본 방향:
- trial: 제한적 또는 locked
- standard: enabled
- pro: enabled
- enterprise: enabled

원칙:
- API 키 등록과 서버사이드 연결이 끝나기 전에는 enabled로 보이면 안 된다.

### 9-2. 아임웹 상품 → 멤버십

현재 단계 기준:
- 고객용 기능이 아니라 운영자 관리 구조
- 고객 화면에서는 hidden 또는 직접 비노출

운영 원칙:
- 운영자 외부 연동 탭에서만 관리
- 실제 webhook / 주문 연동 전에는 연결 준비 또는 미연결

### 9-3. 아임웹 포인트상품 → 크레딧

현재 단계 기준:
- 운영자 관리 구조 우선
- 고객에게는 실제 반영 결과만 간접적으로 보여줄 수 있다

운영 원칙:
- 구매상품 → creditAmount 매핑은 운영자 전용
- 실제 자동 충전 전에는 연결 준비 또는 미연결로 구분

### 9-4. 알림 / 발송 provider

현재 단계 기준:
- placeholder
- provider별 상태 노출 가능

상태 예시:
- 미연결
- 연결 준비
- 테스트 필요
- 연결됨
- 오류

### 9-5. 외부 시장조사 수집

현재 단계 기준:
- placeholder
- 실제 연결 전에는 enabled로 보이면 안 된다

기본 방향:
- trial: hidden 또는 locked
- standard: locked
- pro: 연결 후 enabled 가능
- enterprise: 연결 후 enabled 가능

## 10. 준비중 기능 해제 원칙

준비중 기능은 아래 원칙으로 처리한다.

### 10-1. 전부 해제 금지

- 준비중 기능을 일괄적으로 모두 열지 않는다.
- 실제 연결되지 않은 기능은 구조상 열려 있어도 안 된다.

### 10-2. 실제 연결된 기능만 활성화

- 서버사이드 연결
- provider 설정
- 운영자 매핑
- 최소 검증
이 끝난 기능만 enabled로 전환한다.

### 10-3. 연결 전 단계 표기

연결 전 단계 기능은 아래처럼 상태를 분명히 표기한다.

- 연결 준비
- 운영자 설정 필요
- 권한 필요
- 미연결
- 테스트 필요

### 10-4. 고객 오해 방지

- 고객이 눌렀을 때 “되는 줄 알았는데 안 되는” 구조를 줄인다.
- 설명 없는 잠금보다 이유가 있는 잠금을 우선한다.
- 실제로 동작하지 않으면 동작하는 것처럼 보이게 하지 않는다.

## 11. 운영자 override 원칙

운영자 override는 기본 멤버십 정책을 완전히 무시하는 무질서한 예외 처리가 아니라, 제한적 예외 제어 수단이다.

원칙:
- 기본 정책은 membershipCode 기준으로 간다.
- override는 featureKey + membershipCode 기준 분리 구조를 유지한다.
- 운영자 전용 페이지에서만 조정한다.
- 고객용 화면에서는 override 구조를 직접 노출하지 않는다.
- override 사용 시 왜 예외가 적용되는지 운영자 기준으로 추적 가능해야 한다.


### 11-1. 멤버십 정책 / 기능 노출 / override / 크레딧 운영 역할 구분

운영자 화면에서 아래 4가지는 서로 다른 책임으로 분리한다.

- 멤버십 정책
  - membershipCode 기준의 기본 정책을 관리한다.
  - 등급별 기본 기능 범위, 기본 크레딧 정책, 기본 한도, 기본 권한을 다룬다.
  - 개별 고객 예외 처리나 일시적 기능 개방은 여기서 직접 처리하지 않는다.

- 기능 노출/비노출 제어
  - featureKey 단위의 서비스 노출 상태를 관리한다.
  - enabled / locked / hidden 및 연결 준비 / 미연결 / 테스트 필요 상태를 관리한다.
  - 멤버십 정책의 기본값과 별도로, 현재 단계에서 전체 서비스에 노출할지 여부를 통제한다.

- override
  - 기본 멤버십 정책을 대체하는 것이 아니라 제한적 예외 처리 수단이다.
  - 특정 고객 / 조직 / 기능에 대해 예외를 부여할 때만 사용한다.

- 크레딧 운영
  - 운영자 수동 지급 / 차감 / 조정 / 회수 같은 실제 운영 처리를 담당한다.
  - 아임웹 포인트상품 연동은 자동화 입력 경로일 뿐이며, 수동 운영 기능을 대체하지 않는다.

원칙:
- 기본값은 멤버십 정책에서 관리한다.
- 서비스 전체 롤아웃은 기능 노출/비노출 제어에서 관리한다.
- 개별 예외는 override에서 관리한다.
- 실제 금액성/사용량성 조정은 크레딧 운영에서 관리한다.


### 11-2. 업종별 카드 노출 원칙

운영팀과 영업팀의 카드 세트는 선택 업종 기준으로 다르게 노출한다.

원칙:
- 공통 대메뉴는 유지한다
- 업종에 없는 운영팀 / 영업팀 카드는 기본 노출하지 않는다
- 업종 전환 시 운영팀 / 영업팀 카드 세트가 함께 변경되어야 한다
- 다른 업종용 카드를 같은 화면에 섞어 보여주지 않는다

## 12. 고객용 문구 원칙

잠금과 연결 준비 상태는 문구가 중요하다.

권장 방향:
- 업그레이드 필요
- 운영자 설정 필요
- 연결 준비 중
- 미연결
- 테스트 필요
- 현재 권한으로는 사용할 수 없음

지양할 방향:
- 무조건 “준비중”만 반복
- 실제 연결되지 않았는데 동작하는 것처럼 보이는 문구
- 잠금 이유가 전혀 없는 비활성 버튼

---

## 13. 고객용 화면과 운영자 화면의 구분 원칙

고객용 화면:
- 실제로 쓰는 기능 중심
- 잠금은 최소한 이해 가능하게 표시
- 운영 구조 자체는 최대한 단순하게 보이게 함

운영자 화면:
- 매핑
- 예외 처리
- provider 상태
- 재처리
- override
- 내부 정책
같은 복잡한 제어 구조를 담당

원칙:
- 같은 기능이라도 고객용과 운영자용의 정보 밀도는 달라야 한다.
- 운영자 구조를 고객용 관리자 설정 5탭 안으로 섞지 않는다.

---

## 14. 향후 확장 원칙

향후 아래 항목과 연결 가능해야 한다.

- 저장공간 quota 실제 enforcement
- 저장공간 유료 상품 판매
- 아임웹 주문 기반 자동 멤버십 변경
- 아임웹 주문 기반 자동 크레딧 충전
- 알림/발송 실제 provider 연결
- 엔터프라이즈 조직 단위 권한 제어
- 다점포 / 멀티브랜치별 기능 차등

운영 원칙:
- 현재 단계에서는 구조만 먼저 잡는다.
- 실제 과금과 자동 처리는 외부 연동 안정화 후 확장한다.

## 15. 현재 단계 요약

현재 기능 잠금 운영정책은 아래 상태를 기준으로 본다.
