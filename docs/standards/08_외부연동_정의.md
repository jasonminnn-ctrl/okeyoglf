# 08_외부연동_정의

작성 기준: 2026-03-11  
적용 단계: 공식 8단계 구현 전 기준 / 8-0 준비정리 완료 기준

---

## 1. 문서 목적

이 문서는 OkeyGolf OS의 외부 연동 구조를 실제 운영 SaaS 기준으로 정의하기 위한 문서다.

이 문서의 목적은 다음과 같다.

- OpenAI, 아임웹, 알림/발송 등 외부 시스템 연동 구조를 단계적으로 정리한다.
- 외부 연동 기능이 고객용 화면과 운영자 화면에서 어떻게 분리되어야 하는지 기준을 확정한다.
- 실제 연결된 기능과 placeholder 상태를 혼동하지 않게 한다.
- 상품 구매 → 멤버십 반영, 포인트 상품 구매 → 크레딧 반영 같은 실제 운영 흐름을 먼저 설계한다.
- 외부 연동이 붙어도 기존 저장 구조, 업종 분기, 관리자 통제, 멤버십/크레딧 정책이 흔들리지 않게 한다.

---

## 2. 외부 연동 운영 원칙

외부 연동은 처음부터 모든 기능을 한 번에 완성하지 않는다.

외부 연동은 아래 순서를 따른다.

- 구조 정의
- 운영자 설정 구조
- 서버사이드 호출 구조
- 매핑 구조
- 이력 / 예외 처리 구조
- 실제 API 연결
- 실제 자동 처리
- 고도화

외부 연동 원칙은 다음과 같다.

- API 키는 프론트 코드에 직접 두지 않는다.
- OpenAI 호출은 프론트 직접 호출이 아니라 서버사이드 호출 구조로 처리한다.
- 아임웹은 초기에는 로그인보다 운영 매핑 구조를 우선한다.
- 실제 연결과 placeholder 상태를 명확히 구분한다.
- placeholder를 실제 동작처럼 보이게 만들지 않는다.
- 외부 연동 후에도 관리자 통제 구조를 유지한다.
- 업종 분기, 결과 저장 구조, 멤버십/크레딧 정책과 충돌하지 않게 설계한다.

## 3. 8단계 범위

공식 8단계에서 다루는 외부 연동 범위는 아래와 같다.

- OpenAI 서버사이드 호출 구조
- 운영자 "외부 연동" 탭
- 아임웹 회원 / 주문 / 상품 / 포인트 구조
- 상품 → 멤버십 매핑
- 포인트상품 → 크레딧 매핑
- 알림 / 발송 provider 구조
- 동기화 이력 / 예외 처리 / 수동 재처리 placeholder

이번 단계의 목적은 "완전한 외부 API 연결"보다 아래를 먼저 완성하는 것이다.

- 운영자 화면 구조
- 데이터 구조
- 매핑 구조
- 상태 구조
- 이력 구조
- 예외 처리 구조
- 향후 실제 연결 자리 확보

## 4. 운영자 외부 연동 탭 구조

외부 연동은 고객용 관리자 설정이 아니라 운영자 전용 `/operator` 내부에서 관리한다.

운영자 외부 연동 탭은 최소 아래 섹션으로 구성한다.

### 4-1. 연동 상태

현재 연결 상태를 한눈에 보는 영역이다.

최소 상태 구분:
- 연결됨
- 연결 준비
- 미연결
- 오류
- 테스트 필요

표시 대상:
- OpenAI
- 아임웹 회원/주문
- 상품 → 멤버십 매핑
- 포인트상품 → 크레딧 매핑
- 알림/발송
- 동기화 이력 / 예외 처리

### 4-2. OpenAI 연동

표시 항목 예시:
- provider
- 모델
- API 키 등록 상태
- 서버사이드 호출 준비 상태
- 최근 테스트 여부
- 최근 오류 여부

현재 단계에서는 "연결 준비" 또는 "미설정" 상태 표시가 가능해야 한다.

### 4-3. 아임웹 회원/주문 연동

표시 항목 예시:
- webhook URL 등록 상태
- 최근 동기화 상태
- 최근 이벤트 수신 여부
- 최근 오류 여부
- 회원/주문 연동 준비 상태

현재 단계에서는 실제 호출이 없어도 구조와 상태가 보이게 한다.

### 4-4. 상품 → 멤버십 매핑

아임웹 상품 구매 시 어떤 membershipCode를 부여할지 관리하는 영역이다.

### 4-5. 포인트상품 → 크레딧 매핑

아임웹 포인트상품 구매 시 내부 creditAmount를 어떻게 반영할지 관리하는 영역이다.

### 4-6. 알림 / 발송 연동

provider별 상태를 관리한다.

대상 예시:
- 카카오 알림톡
- SMS
- 이메일

### 4-7. 동기화 이력 / 예외 처리

실패 건, 미매핑 건, 중복 건, 환불 건 등 운영자가 확인하고 재처리할 수 있는 구조를 둔다.

## 5. OpenAI 서버사이드 호출 원칙

OpenAI 관련 처리 원칙은 아래와 같다.

- 프론트엔드는 OpenAI 비밀키를 직접 가지지 않는다.
- OpenAI 호출은 서버사이드 호출 구조를 사용한다.
- 페이지 컴포넌트가 직접 프롬프트 문자열과 비밀키를 함께 다루지 않는다.
- 실제 호출 구조와 연결 준비 상태를 구분한다.
- 호출 로그, 에러, 사용량 추적이 가능한 구조를 둔다.

최소 관리 항목:
- provider
- model
- promptVersion
- requestPayload
- responsePayload
- status
- error
- createdAt

운영 원칙:
- 업종 분기
- 금지 규칙
- 확인 필요 문구
- 프롬프트 버전
이 모두 서버사이드 또는 중앙 관리 구조와 연결될 수 있어야 한다.

## 6. 아임웹 연동 우선순위

아임웹 연동은 로그인보다 운영 매핑이 먼저다.

우선순위는 아래와 같다.

### 1순위
상품 → membershipCode 매핑

설명:
- 아임웹 상품 구매 시 내부 membershipCode를 자동 반영하는 구조를 우선한다.

### 2순위
포인트상품 → creditAmount 매핑

설명:
- 아임웹 포인트상품 구매 시 내부 크레딧을 자동 반영하는 구조를 우선한다.

### 3순위
주문 처리 이력 / 예외 처리

설명:
- 결제 완료, 실패, 중복, 환불 등 실제 운영 이력을 추적할 수 있어야 한다.

### 4순위
회원 매칭

설명:
- 아임웹 회원과 OkeyGolf 내부 조직/계정 연결 구조를 만든다.

### 5순위
실제 로그인 / 인증 연계

설명:
- 아임웹 SSO, 토큰 연계, 로그인 동기화는 후순위다.

운영 원칙:
- 로그인보다 운영 매핑이 먼저다.
- 운영자는 주문 → 멤버십 / 크레딧 흐름을 먼저 이해하고 관리할 수 있어야 한다.

## 6-1. 아임웹 로그인 연동 검토 원칙 보강
현재 OkeyGolf OS의 인증 구조는 Supabase Auth 기반이며,
아임웹 사이트 로그인과 OkeyGolf OS 로그인은 현재 기준으로 분리된 세션이다.

따라서 현 시점에서는 사용자 입장에서 이중 로그인 가능성이 있다.

다만 아임웹 연동의 우선순위는 기존 원칙대로 로그인보다 운영 매핑이 먼저다.

우선순위는 아래와 같이 유지한다.

1. 상품 → membershipCode 매핑
2. 포인트상품 → creditAmount 매핑
3. 주문 처리 이력 / 예외 처리
4. 회원 매칭
5. 실제 로그인 / 인증 연계

추가 원칙:

- 아임웹 회원 연동은 단일 `imweb_member_id` 컬럼 1개로 끝내지 않는다.
- 최소 아래 항목을 구조적으로 관리할 수 있어야 한다.
  - `externalProvider`
  - `externalMemberId`
  - `memberName`
  - `email`
  - `phone`
  - `syncStatus`
  - `lastSyncedAt`
  - `note`
- 현재 Supabase Auth / profiles / user_roles / RLS 구조는 유지한 채 확장하는 방향을 우선한다.
- 아임웹 로그인 연동은 “가능 여부 단정”보다 “확인 필요 항목 분리” 원칙으로 관리한다.
- 최종 고객 대상 완전 SSO 가능 여부는 공식 지원 범위 확인 전까지 확정하지 않는다.
- 초기 권장 방식은 “최초 1회 회원 연결 + 운영 매핑 우선” 구조다.

현재 단계의 다음 1개 작업:

- 아임웹 회원 연동 1차 DB 구조를 추가하여
  회원 매칭 / 동기화 상태 / 운영자 확인 흐름의 기반을 먼저 만든다.

## 6-2. 아임웹 1차 DB 설계 원칙

아임웹 1차 DB 구조는 단일 `profiles.imweb_member_id` 컬럼 추가 방식으로 처리하지 않는다.

공식 1차 구조는 아래 5개 레이어를 기준으로 설계한다.

1. `imweb_site_connections`
2. `imweb_product_membership_mappings`
3. `imweb_point_credit_mappings`
4. `imweb_order_events`
5. `imweb_member_links`

설계 원칙은 다음과 같다.

- 아임웹 연동은 회원 ID 1개 저장보다 사이트 연결 상태와 주문 이벤트 처리 구조가 먼저다.
- 사이트 연결은 `siteCode`, 연결 상태, 최근 동기화 시각, 메모를 관리할 수 있어야 한다.
- 상품 → membershipCode 매핑과 포인트상품 → creditAmount 매핑은 분리된 운영 테이블로 둔다.
- 주문 웹훅 수신 데이터는 원본 payload와 처리 상태를 함께 저장할 수 있어야 한다.
- 주문 구조는 향후 주문 / 주문섹션 / 주문섹션아이템 확장을 고려한다.
- 회원 매칭은 `externalMemberId` 1개만 저장하지 않고 이름, 이메일, 전화번호, syncStatus, lastSyncedAt, note를 함께 관리한다.
- 1차에서는 `profiles.imweb_member_id` 캐시 컬럼을 두지 않는다.
- 로그인 / 인증 연계는 후순위로 유지한다.

## 6-3. 아임웹 상품 분류 원칙 (확정)

OkeyGolf OS의 과금/사용 정책은 **기간형 멤버십 + 사용량형 크레딧** 이중 구조를 따른다.
아임웹 연동 상품은 이 이중 구조에 맞추어 아래 2종으로 분리해서 본다.

1. **멤버십 상품** — 기간형 구독 등급 변경 대상
2. **포인트상품** — 사용량형 크레딧 충전 대상

원칙:
- 멤버십 상품은 `membershipCode` 매핑 대상으로 본다.
- 포인트상품은 `creditAmount` 매핑 대상으로 본다.
- 멤버십 상품은 기간형 구독 등급 변경과 연결한다. 멤버십이 관리하는 범위는 기능 접근 범위, 저장/열람 범위, 스토리지 한도, 운영 도구 사용 범위다.
- 포인트상품은 실제 AI 호출/사용량형 기능에 쓰이는 크레딧 충전 수단으로 본다. 크레딧은 실제 AI 호출이 성공했을 때만 차감한다.
- 저장, 열람, 관리자 설정 조회, 기본 운영 도구 접근, 기본 스토리지 사용은 포인트상품 충전 대상이 아니라 **멤버십 포함 범위**로 본다.
- 상품 분류 기준이 문서로 확정되기 전에는 자동 반영 로직을 구현 완료로 간주하지 않는다.
- **이 정책이 정본 문서에 확정 반영되기 전에는 아임웹 migration을 다음 작업으로 올리지 않는다.**

## 7. 아임웹 회원 연동 구조

아임웹 회원 연동은 최소 아래 항목을 구조적으로 관리할 수 있어야 한다.

- `externalProvider`
- `externalMemberId`
- `memberName`
- `email`
- `phone`
- `syncStatus`
- `lastSyncedAt`
- `note`

운영 원칙:
- 아직 실제 동기화가 없더라도 구조는 먼저 둔다.
- 실제 회원 데이터와 placeholder 상태를 구분한다.
- 운영자 화면에서 상태와 매칭 여부를 확인할 수 있어야 한다.

## 8. 아임웹 주문 / 상품 연동 구조

아임웹 주문 및 상품 연동은 최소 아래 항목을 구조적으로 관리할 수 있어야 한다.

- `externalOrderId`
- `externalProductName`
- `externalProductCode`
- `paidAmount`
- `orderedAt`
- `orderStatus`
- `processedAt`
- `processedBy`
- `syncStatus`
- `syncError`

운영 원칙:
- 주문 단위로 멤버십 반영 / 크레딧 반영 / 예외 처리 이력이 이어져야 한다.
- 중복 주문, 실패 주문, 환불 주문을 구분할 수 있어야 한다.
- 실제 주문 동기화 전에도 운영자 화면에서 구조와 상태를 이해할 수 있어야 한다.

## 9. 상품 → 멤버십 매핑 원칙

아임웹 상품을 내부 membershipCode와 연결하는 구조를 사용한다.

최소 관리 항목:
- `externalProductName`
- `externalProductCode`
- `mappedMembershipCode`
- `mappedMembershipLabel`
- `isActive`
- `note`

예시 방향:
- OkeyGolf Standard → standard
- OkeyGolf Pro → pro
- OkeyGolf Enterprise → enterprise

운영 원칙:
- 고객용 화면에 노출하지 않는다.
- 운영자만 수정할 수 있게 한다.
- membershipCode는 기존 시스템 값을 그대로 사용한다.
- 실제 주문 동기화 후 자동 반영이 가능하도록 설계한다.

---

## 10. 포인트상품 → 크레딧 매핑 원칙

아임웹 포인트상품 또는 충전 상품을 내부 creditAmount와 연결하는 구조를 사용한다.

최소 관리 항목:
- `externalPointProductName`
- `externalPointProductCode`
- `mappedCreditAmount`
- `grantMode`
- `isActive`
- `note`

grantMode 예시:
- `auto`
- `manual_review`
- `manual_only`

운영 원칙:
- 현재 단계에서는 실제 자동 지급이 아직 없어도 구조는 먼저 둔다.
- 운영자는 구매상품 → 크레딧 반영 규칙을 명확히 볼 수 있어야 한다.
- 환불 / 취소 / 중복 주문에 대비한 확장 포인트를 남긴다.

## 11. 외부 처리 이력 구조

외부 연동 후 실제 운영에서는 누가 어떤 주문으로 어떤 멤버십/크레딧을 처리했는지 추적이 가능해야 한다.

최소 관리 항목:
- `historyId`
- `externalProvider`
- `externalOrderId`
- `actionType`
- `appliedMembershipCode`
- `appliedCreditAmount`
- `status`
- `processedAt`
- `processedBy`
- `note`

actionType 예시:
- `membership_sync`
- `credit_grant`
- `refund_revert`
- `manual_override`

운영 원칙:
- 운영자 화면에서 최근 처리 이력을 확인할 수 있어야 한다.
- 실제 처리와 placeholder 상태를 구분한다.

## 12. 예외 처리 / 수동 재처리 원칙

최소 분류 예시:
- `unmapped_product`
- `duplicate_order`
- `sync_failed`
- `refund_pending`

운영 원칙:
- 운영자는 실패 건을 보고 상태를 확인할 수 있어야 한다.
- 운영자 메모를 남길 수 있어야 한다.
- 수동 재처리 placeholder 버튼 또는 구조를 둘 수 있어야 한다.
- 실제 재처리 API는 없어도 구조와 상태는 먼저 둔다.

## 14. 실제 연결 vs placeholder 구분 원칙

외부 연동은 상태 오해가 생기기 쉬우므로 아래 원칙을 반드시 지킨다.

- 실제 연결
  - API 키가 등록되고 실제 서버사이드 호출이 수행되며 응답을 처리하는 상태

- 연결 준비
  - 구조와 UI는 있으나 실제 외부 호출은 아직 없는 상태

- 미연결
  - 설정 자체가 없는 상태

- 테스트 필요
  - 설정은 있으나 실제 정상 동작 여부 검증이 필요한 상태

- 오류
  - 실제 연동은 시도했으나 실패 또는 에러가 발생한 상태

운영 원칙:
- placeholder를 실제 작동처럼 보이게 만들지 않는다.
- 사용자와 운영자가 현재 상태를 오해하지 않게 표시한다.
- 외부 데이터가 없는 상태에서 멤버십 처리 완료, 크레딧 반영 완료, 발송 완료처럼 보이게 만들지 않는다.

---

## 15. 기존 멤버십 / 크레딧 구조와의 연결 원칙

외부 연동은 기존 5단계 구조를 깨지 않아야 한다.

### 15-1. 과금/사용 정책 이중 구조 (확정)

OkeyGolf OS의 과금/사용 정책은 **기간형 멤버십 + 사용량형 크레딧** 이중 구조를 따른다.

- **기간형 멤버십**: 등급별 기능 범위, 저장/열람 범위, 스토리지 한도, 운영 도구 사용 범위를 관리한다.
- **사용량형 크레딧**: 실제 AI 호출이나 사용량형 기능에만 차감한다.
- 저장, 열람, 관리자 설정 조회, 기본 운영 도구 접근, 기본 스토리지 사용은 **멤버십 포함 범위**이며 크레딧 차감 대상이 아니다.

### 15-2. 유지 원칙

- membershipCode는 기존 시스템 값을 그대로 사용한다.
- creditBalance, credit ledger 구조와 충돌 나지 않게 외부 연동 필드를 별도 레이어로 둔다.
- 운영자 수동 지급 / 차감 구조와 연결 가능해야 한다.
- feature access, override, operator guard를 깨지 않는다.

준비중 기능 관련 원칙:
- 준비중 기능을 전부 해제하지 않는다.
- 실제 연결된 기능만 활성화한다.
- 연결 전 단계 기능은 아래 상태로 정리한다.
  - 연결 준비
  - 운영자 설정 필요
  - 권한 필요
  - 미연결

추가 원칙:
- 아임웹 상품 / 포인트상품 연동은 자동 반영 경로이며, 운영자 수동 멤버십 조정이나 수동 크레딧 지급 / 차감 기능을 대체하지 않는다.
- 환불, 중복 주문, 예외 보상, 테스트 지급, CS 조정 같은 운영 상황을 위해 수동 운영 구조는 별도로 유지한다.
- 자동 반영 이력과 수동 조정 이력은 구분해서 추적 가능해야 한다.
- **정책 문서 확정 전에는 아임웹 migration을 다음 작업으로 올리지 않는다.**

## 16. 9단계로 미루는 것

아래 항목은 8단계에서 구조만 고려하고, 실제 고도화는 9단계 이후로 미룬다.

- 운영자 요청 접수함 고도화
- 전담 컨설턴트 내부 메모 / 우선순위 / 결과 업로드
- 고객별 이력 고도화
- 실제 처리 워크플로우 강화
- 저장공간 유료 과금 / 상품 판매
- 아임웹 SSO / 로그인 연계

운영 원칙:
- 8단계와 9단계 범위를 섞지 않는다.
- 8단계는 외부 연동 구조 안정화에 집중한다.
- 9단계는 운영자 처리 고도화에 집중한다.
