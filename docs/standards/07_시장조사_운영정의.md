## 1. 문서 목적

이 문서는 OkeyGolf OS의 시장조사 메뉴를 실제 운영 SaaS 기준으로 정의하기 위한 문서다.

이 문서의 목적은 다음과 같다.

- 시장조사를 단순 입력 폼이 아니라 조사 요청 생성, 최근 작업 확인, 결과 상세 확인, 저장된 결과 연결, 전담 컨설턴트 전환까지 이어지는 실제 워크플로우로 관리한다.
- 외부 수집 엔진이 아직 연결되지 않은 상태에서도 구조와 운영 흐름이 흔들리지 않게 한다.
- 업종별 조사 관점과 예시 질문은 분기하되, 실제 외부 사실이나 KPI 수치를 임의로 생성하지 않는다.
- 향후 8단계 외부 수집 및 외부 연동을 붙일 수 있도록 placeholder 구조를 미리 확보한다.

## 2. 시장조사 운영 원칙

시장조사 메뉴는 독립 메뉴로 유지한다.

시장조사 결과는 저장된 결과와 연결될 수 있어야 한다.

시장조사 결과는 필요 시 전담 컨설턴트 요청으로 전환될 수 있어야 한다.

시장조사 결과는 AI 마케팅팀, AI 운영팀 등 다른 메뉴에서 참고할 수는 있으나, 시장조사 메뉴를 다른 메뉴의 하위 기능으로 종속시키지 않는다.

외부 수집 엔진이 아직 연결되지 않은 상태에서는 현재 결과가 AI 내부 분석 기반 결과임을 명확히 구분해서 표시한다.

실제 외부 조사 결과, 비교 수치, KPI 해석, 시장 데이터, 경쟁사 수치를 지어내지 않는다.

불확실한 내용은 아래와 같은 방식으로 표기한다.

- 확인 필요
- 추가 자료 필요
- 현재 입력 기준 가정
- 내부 분석 기준 제안

## 3. 시장조사 요청 상태 정의

시장조사 요청 상태는 아래 다섯 단계로 관리한다.

- `draft`
  - 의미: 임시 저장
  - 설명: 사용자가 조사 조건을 입력 중이며 아직 실행하지 않은 상태

- `requested`
  - 의미: 조사 요청
  - 설명: 조사 실행을 시작한 상태

- `processing_placeholder`
  - 의미: 처리 중(준비 중)
  - 설명: 외부 수집 연동 전 placeholder 처리 상태

- `completed`
  - 의미: 조사 완료
  - 설명: 현재 단계 기준 AI 내부 분석 결과가 생성된 상태

- `consultant_handoff`
  - 의미: 전담 컨설턴트 전환
  - 설명: 조사 결과를 전담 컨설턴트 처리로 넘긴 상태

운영 원칙은 다음과 같다.

- 현재 단계에서는 `requested`에서 `completed`로 이어지는 흐름이 기본이다.
- `draft`는 향후 임시저장 기능 확장을 위한 상태로 유지한다.
- `processing_placeholder`는 외부 수집 연동 이전 구조 확보용 상태다.
- `consultant_handoff`는 복잡한 조사, 책임이 큰 판단, 추가 검증이 필요한 조사에 사용한다.

## 4. 시장조사 입력 항목 정의

시장조사 요청은 최소 아래 항목을 구조적으로 저장한다.

- `id`
  - 조사 요청 ID

- `createdAt`
  - 생성 시각

- `updatedAt`
  - 수정 시각

- `businessType`
  - 조사 업종

- `region`
  - 조사 지역

- `keyword`
  - 핵심 키워드

- `purpose`
  - 조사 목적

- `scope`
  - 조사 범위

- `count`
  - 조사 수집 개수 또는 조사 대상 수량

- `memo`
  - 추가 메모

- `linkedResultId`
  - 저장된 결과 연결 ID

- `sourceType`
  - 현재 조사 소스 구분

- `collectionStatus`
  - 수집 상태

- `sourceSummary`
  - 수집 또는 분석 기반 요약

- `externalCollectionPlanned`
  - 외부 수집 예정 여부

운영 원칙은 다음과 같다.

- 시장조사 업종은 조사 조건용 선택값이다.
- 회사 기본 업종과 조사 업종은 다를 수 있다.
- 조사 업종 선택이 회사 기본 업종을 자동 변경하지는 않는다.
- 회사 기본 업종과 조사 업종이 다를 경우 안내 문구를 노출할 수 있다.

## 5. 조사형 템플릿 구조

시장조사는 공통 메뉴 구조를 유지하되, 최소 아래 다섯 개 조사형 템플릿으로 분기한다.

- `competitor`
  - 경쟁사 조사
  - 경쟁사 비교, 운영 방식, 포지셔닝, 차별화 포인트 검토

- `area`
  - 지역/상권 조사
  - 입지, 주변 수요, 상권, 고객 유입 관점 조사

- `customer`
  - 고객/수요 조사
  - 고객층, 이용 동기, 재방문, 수요 특성 조사

- `promotion`
  - 프로모션/상품 아이디어 조사
  - 이벤트, 상품, 구성안, 마케팅 아이디어 조사

- `operation`
  - 운영 이슈 조사
  - 운영상 문제, 병목, 관리 포인트, 개선 기회 조사

운영 원칙은 다음과 같다.

- 템플릿은 조사 관점을 정리하는 역할이다.
- 템플릿이 실제 외부 사실을 생성하는 근거가 되어서는 안 된다.
- 같은 템플릿이라도 업종별 예시 질문과 관점은 달라질 수 있어야 한다.

시장조사 템플릿은 공통 구조를 유지하되, 업종별 예시 질문과 활용 관점은 달라져야 한다.

- 피팅샵

  - 회원 재등록
  - 레슨 전환
  - 미방문 회원
  - 시간대 운영 관점 강화

  - 팀수
  - 잔여 타임
  - 채널별 판매
  - 객단가

  - 수강생 유지
  - 강사 운영
  - 교육 상품 구조

  - 상품군

- 업종별 문맥과 관점만 달라지고, 사실성 원칙은 동일하게 유지한다.

## 7. 최근 작업 이력 구조

시장조사 메뉴는 최근 작업 이력을 별도로 확인할 수 있어야 한다.

최근 작업 이력에서 확인 가능한 요소는 다음과 같다.

- 요청 ID
- 조사 업종
- 키워드
- 목적
- 상태 badge
- 생성 시각
- 수정 시각
- 저장 결과 연결 여부
- 전담 컨설턴트 전환 여부

운영 원칙은 다음과 같다.

- 최근 작업은 조사 요청의 진행 흐름을 보여준다.
- 저장된 결과는 완성된 결과물 아카이브 성격이다.
- 최근 작업과 저장된 결과는 역할을 분리한다.

역할 구분은 아래와 같다.

- 최근 작업 = 진행 흐름
- 저장된 결과 = 아카이브 / 재열람

## 8. 조사 결과 상세 운영 기준

시장조사 결과 상세에서는 최소 아래 정보를 읽을 수 있어야 한다.

- 조사 업종
- 입력 조건
- 결과 요약
- 결과 섹션
- 저장 시각
- 수정 시각
- 저장된 결과 연결 여부
- 전담 컨설턴트 전환 이력
- 외부 수집 상태
- 현재 결과가 내부 분석 기반인지 여부

운영 원칙은 다음과 같다.

- 시장조사 결과 상세에서도 다운로드, 공유, 전달 구조가 이어질 수 있어야 한다.
- 단, 실제 외부 발송이나 실제 외부 연동이 없는 경우 placeholder와 실제 동작을 구분한다.
- 결과 상세에서 사용자에게 현재 상태를 오해 없이 보여준다.

## 9. 저장된 결과 삭제 규칙

시장조사 결과는 저장된 결과 구조와 동일한 삭제 규칙을 따른다.

삭제 규칙은 다음과 같다.

- 삭제 위치
  - SavedPage 목록
  - ResultDetailDrawer 상세

- 삭제 확인 문구
  - 제목: 삭제하시겠습니까?
  - 본문: 되돌릴 수 없습니다.

- 확인 시
  - ResultStore에서 제거
  - localStorage에서도 제거

- 취소 시
  - 아무 것도 삭제하지 않음

- 삭제 후
  - 목록
  - 카운트
  - 저장량
  - 최근 결과
  가 즉시 갱신된다.

적용 범위는 다음과 같다.

- generation
- research
- consultant
- manual

주의사항은 다음과 같다.

- 현재 단계에서는 복구 기능이 없다.
- 현재 삭제는 hard delete 성격이다.
- 향후 서버 저장 전환 시 soft delete로 바꾸기 쉬운 구조를 유지한다.

## 10. 저장공간 사용량 추정 구조

현재 단계에서는 localStorage 기반으로 저장공간 사용량을 추정한다.

기본 지표는 다음과 같다.

- `totalSavedCount`
  - 전체 저장 건수

- `totalResearchCount`
  - 시장조사 결과 건수

- `totalGenerationCount`
  - 일반 생성 결과 건수

- `totalConsultantCount`
  - 컨설턴트 결과 건수

- `estimatedStorageBytes`
  - 추정 바이트 수

- `estimatedStorageLabel`
  - 사람이 읽기 쉬운 KB / MB 단위 문자열

운영 원칙은 다음과 같다.

- 현재 저장량은 서버 저장량이 아니라 클라이언트 기준 추정치다.
- plainText, sections, 기본 오버헤드 등을 기준으로 계산한다.
- 사용자에게는 KB 또는 MB 단위로 읽기 쉬운 형태로 표시한다.

멤버십별 soft quota placeholder는 다음과 같이 본다.

- `trial`
  - 20건

- `standard`
  - 100건

- `pro`
  - 500건

- `enterprise`
  - 9999건
  - 무제한 placeholder로 간주

주의사항은 다음과 같다.

- 현재는 soft quota다.
- 강제 차단 로직은 아직 없다.
- 80% 이상 사용 시 경고 표시만 한다.
- 실제 저장공간 과금 및 상품 판매는 후속 단계에서 다룬다.

## 11. localStorage 기반 한계와 서버 전환 포인트

### 현재 한계

- 브라우저별 localStorage 용량 제한이 있다.
- 기기 간 동기화가 되지 않는다.
- 브라우저 초기화 시 데이터가 사라질 수 있다.
- 실제 서버 저장량이 아니라 클라이언트 추정치만 가능하다.

### 향후 서버 전환 포인트

- `ResultStoreContext`의 localStorage 로직을 DB CRUD로 교체한다.
- `deleteResult`를 soft delete 구조로 전환한다.
- 저장공간 사용량을 서버 기준 실제 수치로 계산한다.
- 저장 quota를 실제 enforcement 구조로 연결한다.
- 시장조사 요청 저장도 별도 DB 테이블로 전환한다.

## 12. 외부 수집 placeholder 구조

시장조사는 향후 외부 수집 엔진 연결을 위한 placeholder 구조를 가진다.

핵심 필드는 다음과 같다.

- `sourceType`
  - 현재 단계 값: `ai_internal`
  - 향후 확장: `external_api`, `manual_collection` 등

- `collectionStatus`
  - 현재 단계에서는 내부 처리 완료 기준
  - 향후 실제 수집 상태 반영

- `sourceSummary`
  - 현재는 내부 분석 기반 안내 문구
  - 향후 외부 데이터 출처 요약으로 확장

- `externalCollectionPlanned`
  - 현재는 `false`
  - 향후 외부 수집 트리거 여부 반영

운영 원칙은 다음과 같다.

- 현재 단계에서는 실제 외부 호출이 없다.
- 지금은 구조만 확보한다.
- 사용자가 placeholder를 실제 수집 결과로 오해하지 않게 표시한다.

## 13. 확인 필요 / 가정 기반 제안 원칙

시장조사 결과 생성 시 아래 원칙을 반드시 따른다.

- 확인되지 않은 외부 사실은 단정하지 않는다.
- 비교 수치, 순위, 시장 규모, 경쟁사 수치를 임의 생성하지 않는다.
- 데이터가 없는 KPI 해석을 하지 않는다.
- 필요 시 아래 표현을 사용한다.
  - 확인 필요
  - 추가 자료 필요
  - 현재 입력 기준 가정
  - 내부 분석 기준 제안
- 책임 있는 판단, 금전/계약/운영 리스크가 큰 경우 전담 컨설턴트 검토를 우선한다.

## 14. 다른 메뉴와의 연결 원칙

- 시장조사는 독립 메뉴로 유지한다.
- 저장된 결과와 연결될 수 있어야 한다.
- AI 마케팅팀, AI 운영팀 등은 시장조사 결과를 참고할 수 있다.
- 마케팅팀과의 연결은 약한 연결로 유지한다.
  - 예: 시장조사 결과 참고하기
  - 예: 시장조사 메뉴로 이동하기
- 시장조사 메뉴를 다른 메뉴의 하위 기능으로 종속시키지 않는다.

## 15. 현재 단계 요약

현재 시장조사 메뉴는 아래 수준까지 완료된 것으로 본다.

완료된 항목:
- 조사 요청 저장 구조
- 상태 분기
- 업종별 템플릿 분기
- 최근 작업 이력
- 저장된 결과 연결
- 삭제 기능
- 저장공간 사용량 표시
- 외부 수집 placeholder 구조

후속 단계:
- 실제 외부 수집 엔진 연동
- 실제 서버 저장
- soft delete 전환
- 저장공간 실제 과금 및 상품화
- 아임웹 및 외부 연동과의 실제 연결
