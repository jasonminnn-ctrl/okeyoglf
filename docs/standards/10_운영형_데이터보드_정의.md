# 10_운영형_데이터보드_정의

작성 기준: 2026-03-12

## 1. 문서 목적

이 문서는 OkeyGolf OS에서 AI 답변형 기능과 별도로, 운영자가 직접 입력/수정/조회/필터링/정렬/다운로드할 수 있는 운영형 데이터 화면의 기준을 정의한다.

목적:
- KPI, 고객, 이용권, 상담, 요청, 크레딧 이력 등 운영 데이터를 표형/리스트형으로 관리할 수 있게 한다
- AI 기능에만 의존하지 않고 실제 SaaS형 운영 화면을 보강한다
- 향후 CSV / XLSX export, 대량 수정, 필터링, 브랜치별 집계 구조의 기준을 미리 잡는다

## 2. 기본 원칙

- AI가 처리할 수 있는 일은 AI가 처리한다
- 사용자가 직접 관리해야 하는 운영 데이터는 표형 화면으로 제공한다
- 저장된 결과와 운영형 데이터는 연결될 수 있어야 한다
- 고객용 화면과 운영자용 화면의 편집 권한은 분리한다
- 업종별 운영 데이터 항목은 다를 수 있다

## 3. 운영형 데이터보드 대상

- KPI 보드
- 회원/고객 리스트
- 이용권/레슨 현황
- 상담/재등록/미방문 관리
- 요청 접수함 / 처리 이력
- 크레딧 ledger
- 멤버십 변경 이력
- 외부 연동 처리 이력

추가 대상:
- 저장 결과 운영 보드
- 시장조사 운영 보드
- KPI 차트 보드
- 운영 점검 보드 (`assistant_tasks`)
- 캠페인 보드 (`assistant_campaigns`)
- 일정/마감 보드 (`assistant_reminders`)
- 체크리스트 실행 보드 (`assistant_checklists` + `assistant_checklist_items`)

### 운영형 보드 공통 컬럼 원칙
- 제목
- 업종/조직
- 분류
- 상태
- 우선순위
- 담당자
- 기한 또는 적용기간
- source_type (ai_generated / user_created / imported)
- linked_result_id
- 메모
- 최근 수정일


## 4. 공통 기능

- 검색
- 필터
- 정렬
- 상태 배지
- 행 상세 보기
- 수정 이력
- 다운로드
- CSV export
- XLSX export
- 권한별 편집/조회 구분

추가 기능:
- 차트 시각화
- 집계 카드
- 브랜치/조직/업종 필터 공통화
- 표와 차트의 상호 연결


## 5. KPI 보드 원칙

- KPI는 AI 해석 화면과 별도로 표형 관리 보드가 있을 수 있다
- 목표값 / 현재값 / 기준기간 / 담당자 / 메모를 분리한다
- 외부 연동 데이터와 수기 입력 데이터를 구분한다
- 데이터 없는 KPI는 해석을 단정하지 않는다

추가 구조:
- KPI 목표값
- KPI 현재값
- 기준 기간
- 담당자
- 입력 방식(수기 / 외부 연동 / 계산값)
- 메모
- 최근 변경일

추가 원칙:
- KPI는 AI 해석 화면과 별도로 표형 관리가 가능해야 한다
- 데이터가 없는 KPI는 빈값으로 유지하고 억지 계산하지 않는다
- 업종별 KPI 템플릿을 다르게 둘 수 있다


## 6. 크레딧 운영 보드 원칙

- 조직/고객 검색 가능
- 현재 잔액 표시
- 수동 지급 / 차감 가능
- 조정 사유 필수
- 자동 반영 이력과 수동 조정 이력 분리

## 6-A. 클릭 이동 / 상세 열기 원칙

운영형 데이터보드와 최근 활동 영역은 dead click 없이 실제 이동 또는 상세 열기를 제공해야 한다.

기본 원칙은 다음과 같다.

- 대시보드의 최근 생성 결과는 클릭 시 상세 결과로 이동하거나 상세 Drawer를 연다.
- 최근 시장조사 결과는 클릭 시 해당 결과 상세로 이동하거나 상세 Drawer를 연다.
- 최근 컨설턴트 요청은 클릭 시 해당 요청 상세로 이동하거나 상세 Drawer를 연다.
- 크레딧 사용 내역은 관련 결과가 있을 경우 클릭 시 해당 결과 상세로 이동할 수 있어야 한다.
- 우측 하단 저장 완료 알림은 클릭 또는 액션 버튼을 통해 방금 저장한 결과를 열 수 있어야 한다.
- 클릭 불가능한 상태라면 hover/버튼/링크처럼 오해되게 만들지 않는다.

## 7. 다운로드 원칙

- 결과형 산출물: PDF / 문서 / PPT
- 데이터형 산출물: CSV / XLSX
- 사용자에게 현재 어떤 형식으로 내려가는지 명확히 표시한다

추가 원칙:
- CSV는 기본 데이터 내보내기 형식으로 유지한다
- XLSX는 서식형 데이터 내보내기 형식으로 추가한다
- export는 현재 화면의 필터 / 정렬 / 선택 상태와 최대한 일치해야 한다
- 결과형 산출물과 데이터형 산출물의 다운로드 방식을 혼동하지 않게 구분한다


추가 우선순위:
- 9단계 잔수정에서는 운영형 리스트 데이터의 CSV export를 먼저 반영할 수 있다
- XLSX export는 10단계의 고급 데이터보드 확장 범위로 둔다
- export는 현재 화면 필터/정렬 기준 데이터와 최대한 일치해야 한다


## 8. 단계 연결

- 9단계: 운영자 콘솔 내 표형 관리 화면 기초 구조
- 10단계: KPI 보드, 브랜치별 집계, 고급 export, 엔터프라이즈 확장

추가 원칙:
- 현재 단계의 데이터보드는 Context 또는 localStorage 기반 mock/임시 저장 구조를 사용할 수 있다
- 다음 단계에서는 DB / Storage 실연동으로 전환하기 쉬운 형태를 우선한다
- localStorage 기반 데이터는 최종 구조가 아니라 실연동 전환을 위한 임시 운영 구조로 본다
