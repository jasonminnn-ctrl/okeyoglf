# 10_운영형_데이터보드_정의

최종 갱신: 2026-03-14

## 1. 문서 목적

이 문서는 OkeyGolf OS에서 AI 답변형 기능과 별도로, 운영자가 직접 입력/수정/조회/필터링/정렬/다운로드할 수 있는 운영형 데이터 화면의 기준을 정의한다.

목적:
- KPI, 고객, 이용권, 상담, 요청, 크레딧 이력 등 운영 데이터를 표형/리스트형으로 관리할 수 있게 한다
- AI 기능에만 의존하지 않고 실제 SaaS형 운영 화면을 보강한다
- CSV / XLSX export, 대량 수정, 필터링, 브랜치별 집계 구조의 기준을 미리 잡는다

## 2. 기본 원칙

- AI가 처리할 수 있는 일은 AI가 처리한다
- 사용자가 직접 관리해야 하는 운영 데이터는 표형 화면으로 제공한다
- 저장된 결과와 운영형 데이터는 연결될 수 있어야 한다
- 고객용 화면과 운영자용 화면의 편집 권한은 분리한다
- 업종별 운영 데이터 항목은 다를 수 있다

## 3. 운영형 데이터보드 대상

### 3-1. 운영자 콘솔 보드 (/operator)
- KPI 보드
- 저장 결과 운영 보드
- 시장조사 운영 보드
- 크레딧 ledger
- 컨설턴트 요청 관리
- 멤버십 / override 관리
- 외부 연동 관리

### 3-2. 고객용 운영형 보드 (AI 비서 하위)
- 운영 점검 보드 (`assistant_tasks`) — `/ai-assistant/ops-check`
- 캠페인 보드 (`assistant_campaigns`) — `/ai-assistant/campaign-planner`
- 일정/마감 보드 (`assistant_reminders`) — `/ai-assistant/reminder-board`
- 체크리스트 실행 보드 (`assistant_checklists` + `assistant_checklist_items`) — `/ai-assistant/checklist-manager`

### 현재 구현 상태 (2026-03-13)
- 운영자 콘솔: 전체 탭 DB 연동 완료 (KPI, 결과, 시장조사, 크레딧, 컨설턴트, 조직, 멤버십, 외부연동, 공지, 운영 권장)
- 고객용 4종 운영형 보드: DB 연동 완료, 하이브리드 패턴(직접입력 + AI 비서 + export + 추적 + 상세보기) 적용
- CSV/XLSX export: 운영자 콘솔 + 고객용 4종 모두 실제 동작
- 공통 source 분류 탭 (`OperationalSourceTabs`): 4종 운영형 페이지에 전체/직접등록/AI제안/운영권장 탭 통일 적용
- 상세 보기: `OperationalDetailDialog` + `OperationalAttachmentSection` 적용
- 운영 권장 공급: `operator_recommendations` → `RecommendationSupplyPanel` 자동 공급 (4종 페이지 연결 완료)
- 정밀 타겟팅: target_org_id / target_branch_code 클라이언트 필터링 적용 완료
  - target_org_id NULL = 전체 공통, 값 있으면 해당 조직만
  - target_branch_code NULL = 전체 사업장, 값 있으면 해당 사업장만
  - 한계: branch_code 조회는 organizations 테이블 연동 시 완성 (현재 null → 전체 매칭)

### 운영형 보드 공통 컬럼 원칙

모든 운영형 보드가 공유하는 기본 컬럼:
- 제목 (title)
- 상태 (status)
- source_type (ai_generated / user_created / ops_recommended)
- linked_result_id
- 메모/상세내용 (memo / description)
- 생성일 (created_at)
- 최근 수정일 (updated_at)
- org_id

완료 추적 컬럼 (체크리스트/운영점검 등 완료형 객체):
- 담당자 (assignee_name)
- 완료자 사용자 ID (completed_by_user_id)
- 완료자 이름 (completed_by_name)
- 완료시각 (completed_at)

운영 점검 전용:
- risk_source (user_created / ai_suggested / ops_recommended)

첨부 파일:
- `operational_attachments` 테이블: entity_type / entity_id / file_path / file_name / mime_type / uploaded_by
- Supabase Storage `operational-attachments` 버킷 연동

### 운영 권장 관리 (`operator_recommendations`)
- recommendation_type (ops_recommended / risk_signal / doing_well / must_check / weekly_action)
- target_business_types (업종 배열)
- target_org_id (조직 타겟팅)
- target_branch_code (사업장 타겟팅)
- target_user_ids (사용자 타겟팅 — uuid 배열, 비어있으면 조직/사업장 전체)
- category (ops_check / campaign / reminder / checklist / general)

#### 타겟팅 계층 (5단계)
1. 전체 공통: target_org_id=NULL, target_user_ids=[]
2. 업종 공통: target_business_types에 업종 지정
3. 조직 전용: target_org_id 지정
4. 사업장 전용: target_branch_code 지정
5. 사용자 전용: target_user_ids에 사용자 UUID 지정

#### 메뉴별 운영 권장 현황 보드 (`RecommendationCoverageBoard`)
- 카테고리별 등록 개수
- 유형별(recommendation_type) 개수
- 대상 분포(전체 공통 / 조직 전용 / 사업장 전용 / 사용자 전용)
- 활성 / 비활성 / 기간 종료 항목 수
- 현재 공급 중인 항목 리스트

#### DB 반영 상태 (2026-03-14)
- `assistant_tasks`: assignee, completed_by_user_id, completed_by_name, completed_at, risk_source 모두 DB 컬럼 존재
- `assistant_checklist_items`: assignee_name, completed_by_user_id, completed_by_name, checked_at 모두 DB 컬럼 존재
- `assistant_campaigns`: source_type, linked_result_id 존재, 완료 추적 컬럼은 캠페인 특성상 미적용
- `assistant_reminders`: source_type, linked_result_id 존재, 완료 추적 컬럼은 리마인드 특성상 미적용
- `operator_recommendations`: target_user_ids 컬럼 추가 완료 (2026-03-14), RLS 적용, 운영자 전용 CRUD
- `operational_attachments`: entity_type/entity_id 기반 다형적 첨부 구조

## 5. KPI 보드 원칙

- KPI는 AI 해석 화면과 별도로 표형 관리 보드가 있을 수 있다
- 목표값 / 현재값 / 기준기간 / 담당자 / 메모를 분리한다
- 외부 연동 데이터와 수기 입력 데이터를 구분한다
- 데이터 없는 KPI는 해석을 단정하지 않는다

추가 구조:
- KPI 목표값
- KPI 현재값
- 기준 기간
- 담당자
- 입력 방식(수기 / 외부 연동 / 계산값)
- 메모
- 최근 변경일

추가 원칙:
- KPI는 AI 해석 화면과 별도로 표형 관리가 가능해야 한다
- 데이터가 없는 KPI는 빈값으로 유지하고 억지 계산하지 않는다
- 업종별 KPI 템플릿을 다르게 둘 수 있다

## 6-A. 클릭 이동 / 상세 열기 원칙

운영형 데이터보드와 최근 활동 영역은 dead click 없이 실제 이동 또는 상세 열기를 제공해야 한다.

기본 원칙은 다음과 같다.

- 대시보드의 최근 생성 결과는 클릭 시 상세 결과로 이동하거나 상세 Drawer를 연다.
- 최근 시장조사 결과는 클릭 시 해당 결과 상세로 이동하거나 상세 Drawer를 연다.
- 최근 컨설턴트 요청은 클릭 시 해당 요청 상세로 이동하거나 상세 Drawer를 연다.
- 크레딧 사용 내역은 관련 결과가 있을 경우 클릭 시 해당 결과 상세로 이동할 수 있어야 한다.
- 우측 하단 저장 완료 알림은 클릭 또는 액션 버튼을 통해 방금 저장한 결과를 열 수 있어야 한다.
- 클릭 불가능한 상태라면 hover/버튼/링크처럼 오해되게 만들지 않는다.

## 7. 다운로드 원칙

- 결과형 산출물: PDF / 문서 / PPT (서버 생성 필요, 현재 미구현)
- 데이터형 산출물: CSV / XLSX (구현 완료)
- 사용자에게 현재 어떤 형식으로 내려가는지 명확히 표시한다

추가 원칙:
- CSV는 기본 데이터 내보내기 형식으로 유지한다
- XLSX는 서식형 데이터 내보내기 형식으로 추가한다
- export는 현재 화면의 필터 / 정렬 / 선택 상태와 최대한 일치해야 한다
- 결과형 산출물과 데이터형 산출물의 다운로드 방식을 혼동하지 않게 구분한다

### 문서형 vs 데이터형 export 구분
- **문서형 export** (saved_results): TXT 즉시 다운로드 구현 완료, PDF/DOC/PPT는 서버 생성 필요 (미구현)
- **데이터형 export** (운영 객체): CSV/XLSX 구현 완료 (4종 운영형 페이지 + 운영자 콘솔)
- 이 두 가지를 섞지 않는다
