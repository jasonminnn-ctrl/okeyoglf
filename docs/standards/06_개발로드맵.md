# 06_개발로드맵

최종 갱신: 2026-03-14

## 1. 문서 목적
이 문서는 OkeyGolf OS를 러버블 중심으로 단계별 구현하기 위한 개발 로드맵 문서다.
목표는 한 번에 모든 기능을 억지로 붙이는 것이 아니라, 구조를 먼저 고정한 뒤 기능, 저장, AI, 크레딧, 다운로드, 외부 연동을 순차적으로 붙여서 실제 상용 가능한 SaaS로 고도화하는 것이다.

---

## 2. 전체 개발 원칙

### 2-1. 한 번에 다 구현하지 않는다
처음부터 로그인, AI, 크레딧, PDF, 카카오 공유, 시장조사 수집 엔진, 외부 연동을 한 번에 붙이면 구조가 꼬일 가능성이 매우 높다.
반드시 단계별로 구현한다.

### 2-2. 구조를 먼저 고정한다
대메뉴, 업종 분기, 관리자 설정, 저장된 결과, 전담 컨설턴트 구조를 먼저 고정한다.

### 2-3. 기존 화면은 최대한 재사용한다
이미 만들어둔 대시보드와 생성기 페이지는 삭제하지 않고 새 구조 아래에 재배치한다.

### 2-4. 관리자 통제 구조를 초반부터 고려한다
프롬프트, 지식베이스, 출력 기준, 금지 규칙, 기능 노출, 멤버십 잠금, 크레딧 정책 등은 초반 구조에서 빠지면 안 된다.

### 2-5. 실제 상용화 기준으로 본다
데모가 아니라 실제 구독형 SaaS를 목표로 한다.

## 4. 8단계 외부 연동 상세 상태

### 구현 완료
- 아임웹 5개 DB 테이블 (site_connections, product_membership_mappings, point_credit_mappings, order_events, member_links)
- `/operator` 외부 연동 탭 UI (연동 상태, OpenAI, 아임웹, 매핑, 알림, 이력)
- repository 연결 및 실데이터 렌더링 검수
- 과금/사용 정책 문서 확정 (기간형 멤버십 + 사용량형 크레딧)

### 실연동 대기 (부분 구현)
- OpenAI: 에지 함수 구조 존재, 실제 API 호출 미연동 (mock 응답)
- 아임웹: DB 구조 완료, 실제 webhook 수신 미연동 (테스트 연결 단계)
- 알림/발송: provider 구조만, 실제 발송 미연동

### 미구현
- 아임웹 실제 webhook 자동 처리
- 자동 멤버십/크레딧 반영
- 카카오/SMS/이메일 실제 발송
- 회원 매칭 자동화

## 5. 현재 우선순위 (2026-03-13)

### 우선순위 1: 실제 AI 호출 연동
- mock 응답을 실제 LLM 호출로 전환
- AI 비서 하단 패널의 규칙 기반 파싱을 LLM 기반 의도 분석으로 고도화
- 크레딧 차감을 실제 호출 성공 시점에 연동

### 우선순위 2: 멤버십 차등 노출
- featureKey별 멤버십 등급 기반 실제 enabled/locked/hidden 적용
- AI 제안 리스크 / 운영 권장 리스크의 멤버십별 노출 차등

### 우선순위 3: 아임웹 실연동 1차
- webhook 수신 함수
- 주문 → 멤버십/크레딧 자동 반영
- 테스트 연결 → 운영 연결 전환

### 우선순위 4: 제품 진실성 반영
- 외부연동 상태를 `연결 준비 / 테스트 연결 / 운영 연결 / 오류`로 구분
- placeholder와 실연동 상태를 분리

### 우선순위 5: 기술 정리
- `types.ts` 재생성
- `imweb-repository.ts`의 `as any` / `as unknown` 캐스팅 정리
- 정책 문서와 UI 문구 최종 교차 검수

### Phase 9-3: 운영 권장 공급 구조 + 공통 source 탭 공통화 1차 (2026-03-14 완료)
- `OperationalSourceTabs` 공통 컴포넌트 — 4종 운영형 페이지에 전체/직접등록/AI제안/운영권장 탭 통일 적용
- `filterBySourceTab()` 헬퍼 — source_type / risk_source 기준 공통 필터링
- `/operator` 운영 권장 관리 탭 신설 (`OperatorRecommendationTab`)
- `operator_recommendations` DB 테이블 생성 (유형/업종/조직/사업장 타겟팅, RLS 적용)
- 운영 권장 유형: 운영 권장 / 위험신호 / 잘하고 있는 점 / 꼭 확인할 것 / 이번 주 권장 액션

### Phase 10 (다음): 운영 권장 고객 자동 공급 + 직원 업무 리스트
- operator_recommendations → 고객 운영형 페이지 자동 노출 연동
- 내 업무 보기 / 담당자 필터 / 직원별 업무 리스트
- AI 비서 실제 LLM 호출 + 크레딧 차감 연동

## 6. 단계별 프롬프트 운영 원칙
- 한 프롬프트에 너무 많은 요구를 넣지 않는다
- 각 단계가 끝나면 UI와 구조를 먼저 검수한다
- 검수 후 다음 프롬프트를 넣는다
- 구조가 흔들리면 이전 문서 기준으로 다시 고정한다
- 중요한 결정은 문서화하고 프로젝트 소스로 축적한다

## 8. 구현 시 주의사항
- 초기부터 모든 걸 붙이지 않는다
- 관리자 설정 없이 AI 기능만 먼저 고도화하지 않는다
- 저장된 결과 구조 없이 출력만 늘리지 않는다
- 업종 분기를 무시하고 공통 답변형으로 흘러가게 두지 않는다
- 전담 컨설턴트 흐름을 단순 문의 페이지로 축소하지 않는다
- 마케팅팀과 디자인팀은 계속 분리 유지한다
- AI 경영지원은 백오피스 실무 기준으로 본다
