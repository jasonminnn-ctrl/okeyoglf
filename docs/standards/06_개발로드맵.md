# 06_개발로드맵

## 1. 문서 목적
이 문서는 OkeyGolf OS를 러버블 중심으로 단계별 구현하기 위한 개발 로드맵 문서다.
목표는 한 번에 모든 기능을 억지로 붙이는 것이 아니라, 구조를 먼저 고정한 뒤 기능, 저장, AI, 크레딧, 다운로드, 외부 연동을 순차적으로 붙여서 실제 상용 가능한 SaaS로 고도화하는 것이다.

이 문서는 향후 아래 기준이 된다.
- 러버블 프롬프트 입력 순서
- 각 단계별 구현 목표
- 단계별 검수 포인트
- 나중에 붙일 기능과 현재 보류할 기능 구분
- 운영자/관리자 구조 확장 계획
- AI/지식베이스/크레딧/외부 연동 구현 순서

---

## 2. 전체 개발 원칙

### 2-1. 한 번에 다 구현하지 않는다
처음부터 로그인, AI, 크레딧, PDF, 카카오 공유, 시장조사 수집 엔진, 외부 연동을 한 번에 붙이면 구조가 꼬일 가능성이 매우 높다.
반드시 단계별로 구현한다.

### 2-2. 구조를 먼저 고정한다
대메뉴, 업종 분기, 관리자 설정, 저장된 결과, 전담 컨설턴트 구조를 먼저 고정한다.

### 2-3. 기존 화면은 최대한 재사용한다
이미 만들어둔 대시보드와 생성기 페이지는 삭제하지 않고 새 구조 아래에 재배치한다.

### 2-4. 관리자 통제 구조를 초반부터 고려한다
프롬프트, 지식베이스, 출력 기준, 금지 규칙, 기능 노출, 멤버십 잠금, 크레딧 정책 등은 초반 구조에서 빠지면 안 된다.

### 2-5. 실제 상용화 기준으로 본다
데모가 아니라 실제 구독형 SaaS를 목표로 한다.
따라서 로그인/권한, 저장, 재열람, 요청 흐름, 운영자 관리, 전달 구조까지 고려한다.

## 4. 단계별 상세 로드맵

## 4-1. 1단계: 제품 구조 확정 및 메뉴 재편
### 목표
- 현재 화면을 바탕으로 대메뉴 구조를 실무형으로 재편
- 기존 생성기 페이지를 새 메뉴 구조 아래 재배치
- 공통 메뉴 + 내부 업종 분기 구조 고정
- 관리자 설정의 큰 틀 반영

### 구현 대상
- 대시보드
- AI 비서
- AI 운영팀
- AI 영업팀
- AI 마케팅팀
- AI 디자인팀
- AI 경영지원
- 시장조사
- 전담 컨설턴트
- 저장된 결과
- 관리자 설정

### 이 단계에서 해야 할 것
- 랜딩 페이지 구성
- 카드 구조 구성
- 전담 컨설턴트 CTA 삽입
- 운영자/엔터프라이즈 전용 영역 placeholder
- 죽은 메뉴 제거

### 이 단계에서 하지 않을 것
- 실제 AI 연동
- 크롤링
- 크레딧 차감
- PDF 다운로드
- 카카오 공유
- 외부 회원 연동

### 검수 포인트
- 사이드바 구조가 맞는가
- 각 메뉴에 landing page가 있는가
- 각 메뉴에 전담 컨설턴트 CTA가 있는가
- 관리자 설정이 확장형 구조로 보이는가

---

## 4-1A. 1.5단계: 고객 기본정보 및 업종별 운영기준 입력 구조

### 목표
- 고객용 관리자 설정과 오키골프 내부 운영 통제 영역을 분리
- 고객사가 프롬프트를 만지는 구조가 아니라 운영 컨텍스트를 입력하는 구조로 전환
- 업종 분기에 필요한 기본 컨텍스트 입력 폼을 먼저 준비

### 구현 대상
- 고객용 관리자 설정 단순화
- 조직 기본정보 입력 구조
- 업종 및 운영 형태 설정 구조
- 업종별 운영 기본정보 입력 폼
- 브랜드/자료 업로드 placeholder
- 내부 실무 기준정보 입력 구조
- 운영자 전용 관리 내 정책/프롬프트/엔진 운영 구조 보강
- 대시보드 및 각 팀 메뉴의 현재 적용 기준 placeholder

### 검수 포인트
- 고객용 관리자 설정에서 프롬프트/금지 규칙/AI 참조 정책이 빠졌는가
- 고객용 관리자 설정에 회사 기본정보/업종별 운영 기본정보가 들어갔는가
- 운영자 전용 관리에 정책 통제 항목이 모였는가
- 이후 2단계 업종 분기를 위한 기본 컨텍스트 구조가 보이는가

---

## 4-2. 2단계: 팀 메뉴 내부 카드/서브도구 정리 및 업종 분기 반영
### 목표
- 각 팀 메뉴를 실제 실무 카드 중심으로 구체화
- 업종별 예시, KPI, 프리셋, 추천 액션 반영
- 이 단계는 1.5단계에서 업종 컨텍스트 입력 구조가 준비된 이후 진행한다.

### 구현 대상
- AI 비서 내부 카드
- AI 운영팀 세부 카드
- AI 영업팀 세부 카드
- AI 마케팅팀 세부 카드
- AI 디자인팀 세부 카드
- AI 경영지원 세부 카드
- 업종별 KPI 카드 구조
- 업종 선택에 따른 내부 분기

### 검수 포인트
- 업종 선택 시 내부 카드/예시가 달라지는가
- 기존 생성기 페이지가 하위도구로 잘 흡수됐는가
- 팀별 역할이 서로 겹치지 않는가

---

## 4-3. 3단계: 저장된 결과 / 첨부 / 재열람 구조
### 목표
- 생성된 결과와 요청 결과를 저장하고 다시 열람하는 구조 확보
- 나중에 AI 결과와 전담 컨설턴트 결과를 함께 관리할 기반 마련

### 구현 대상
- 저장된 결과 목록
- 저장된 결과 상세 페이지
- 카테고리 분류
- 상태값
- 메모
- 재열람 구조
- 향후 복제/재생성 placeholder
- 첨부 placeholder
- 전달 이력 placeholder

### 검수 포인트
- 저장 구조가 모듈별로 분리되는가
- 결과 상세 페이지가 있는가
- 시장조사/전담 요청 결과도 같은 저장 구조로 흡수 가능한가

---

## 4-4. 4단계: AI 생성 구조 및 프롬프트 중앙관리 구조
### 목표
- 실제 AI 기능을 붙일 수 있도록 중앙 관리 구조 설계
- 프롬프트를 하드코딩하지 않고 운영자 전용 관리에서 관리할 기반 마련
- 고객용 관리자 설정과 운영자 전용 관리의 권한 경계를 이 단계부터 분리

### 구현 대상
- 모듈별 프롬프트 구조
- 업종별 추가 지침 구조
- 출력 기준 구조
- 금지 규칙 구조
- 전담 컨설턴트 전환 규칙 구조
- AI 서비스 layer placeholder
- 운영자 전용 관리 내 프롬프트/정책 관리 화면

### 실제 기술 방향
- OpenAI 서버사이드 호출 구조
- 프롬프트는 DB 또는 설정 테이블에서 관리
- 페이지는 직접 프롬프트를 들고 있지 않음

### 검수 포인트
- 모듈별 프롬프트 관리가 가능한 구조인가
- 업종별 분기 규칙을 넣을 수 있는가
- 금지 규칙과 확인 필요 문구를 제어할 수 있는가

---

## 4-5. 5단계: 크레딧 / 멤버십 / 기능 잠금 구조
### 목표
- 구독형 SaaS 운영을 위한 과금/권한 기반 구조 마련
- 기능 접근 제어와 업그레이드 유도를 시스템화

### 구현 대상

- 멤버십 등급
- 기능 잠금
- 전담 컨설턴트 요청 잠금
- 크레딧 정책 placeholder
- 크레딧 잔액/사용 내역 화면
- 운영자 수동 지급/차감 기능
- 운영자 전용 관리 내 기능 노출/비노출 편집
- 메뉴별 노출/비노출 제어
- 카드별 노출/비노출 제어
- 결과 액션별 노출/비노출 제어
- 전담 컨설턴트 전환 버튼 노출/비노출 제어
- 조직별 override 관리 구조
- 고객용 관리자 설정 내 이용현황 탭
- 현재 멤버십 / 업그레이드 안내 / 크레딧 잔액 / 최근 사용내역 / 기능 상태 요약 조회 구조

### 6단계 착수 전 선행 조건

- 고객용 사이드바에는 기존 11개 대메뉴만 노출되어야 한다
- 운영자 관리(/operator), 엔터프라이즈(/enterprise)는 role 기반으로 숨김 및 접근 제어되어야 한다
- 저장된 결과 / 결과 상세 / 시장조사 저장 구조는 실제 저장 store 기준으로 연결되어야 한다
- GenerationFlow 및 시장조사 결과 저장은 toast나 예시 데이터가 아니라 중앙 저장 구조로 연결되어야 한다
- 고객용 관리자 설정 입력값은 중앙 조직 프로필 구조를 통해 AI 생성 컨텍스트에 반영되어야 한다
- 기능 노출 override는 featureKey 단독 기준이 아니라 membershipCode를 포함한 구조로 분리되어야 한다




### 멤버십 예시
- trial
- standard
- pro
- enterprise

### 검수 포인트

- 잠금 기능이 UI상 자연스러운가
- 업그레이드 유도 문구가 명확한가
- 전담 요청 CTA와 멤버십이 연결되는가
- 운영자 전용 관리에서 기능 노출/비노출을 실제 편집할 수 있는가
- 고객용 관리자 설정에는 운영자 제어 항목이 노출되지 않는가
- 고객용 관리자 설정에서 현재 멤버십, 크레딧 잔액, 최근 사용내역을 조회 전용으로 확인할 수 있는가

## 4-6. 6단계: 다운로드 / 공유 / 전달 구조
### 목표
- 결과물을 단순 화면 출력이 아니라 전달 가능한 결과물로 전환
- 향후 PDF, 공유링크, 카카오 공유 확장을 위한 구조 마련

### 구현 대상
- 다운로드 버튼 구조
- 공유 버튼 구조
- 전달 이력 구조
- 저장된 결과 상세에서 내보내기 placeholder
- PDF/문서/PPT용 포맷 분기

### 검수 포인트

- 잠금 기능이 UI상 자연스러운가
- 업그레이드 유도 문구가 명확한가
- 전담 요청 CTA와 멤버십이 연결되는가
- 운영자 전용 관리에서 기능 노출/비노출을 실제 편집할 수 있는가
- 고객용 관리자 설정에는 운영자 제어 항목이 노출되지 않는가
- 고객용 관리자 설정은 회사/업종, 운영정보, 사용자/권한, 기능/브랜드 구조로 정리되어 있는가


---

## 4-7. 7단계: 시장조사 확장 / 외부 수집 구조
### 목표
- 시장조사를 입력 폼 수준에서 실제 조사 워크플로우로 확장
- 외부 데이터 수집 전에도 조사 구조를 안정화

### 구현 대상
- 시장조사 요청 저장
- 최근 작업 이력
- 조사 결과 상세
- 조사형 산출물 구조
- 간단한 조사 템플릿
- 향후 수집 엔진 연결 placeholder

### 검수 포인트
- 조사 요청이 저장되는가
- 조사 결과가 저장된 결과로 연결되는가
- 복잡한 요청은 전담 컨설턴트로 넘길 수 있는가

### 7단계 실제 반영 결과 (2026-03-11 기준)

7단계에서는 시장조사를 단순 입력 폼이 아니라 실제 조사 워크플로우 구조로 확장했다.

반영 완료 항목:
- 시장조사 요청 상태 정의 완료
  - draft
  - requested
  - processing_placeholder
  - completed
  - consultant_handoff
- 조사형 템플릿 5종 구조 완료
  - 경쟁사 조사
  - 지역/상권 조사

### 과금/사용 정책 확정 원칙 (2026-03-13 확정)

OkeyGolf OS의 과금/사용 정책은 **기간형 멤버십 + 사용량형 크레딧** 이중 구조로 확정되었다.

- 멤버십은 등급별 기능 범위, 저장/열람 범위, 스토리지 한도, 운영 도구 사용 범위를 관리한다.
- 크레딧은 실제 AI 호출이나 사용량형 기능에만 차감한다.
- 저장, 열람, 관리자 설정 조회, 기본 운영 도구 접근, 기본 스토리지 사용은 멤버십 포함 범위다.
- 아임웹 멤버십 상품은 `membershipCode` 매핑 대상, 포인트상품은 `creditAmount` 매핑 대상이다.
- **이 정책이 정본 문서(`09_기능잠금_운영정책.md`, `08_외부연동_정의.md`)에 확정 반영되기 전에는 아임웹 migration을 다음 작업으로 올리지 않는다.**

현재 기준 다음 우선순위:

1. 과금/사용 정책 정본 문서 확정 ← **현재 작업 (완료)**
2. 아임웹 1차 DB 설계 문서(`docs/PHASE_8_DB_DESIGN.md`) 확정
3. 아임웹 상품 → membershipCode 매핑 migration
4. 아임웹 포인트상품 → creditAmount 매핑 migration
5. 주문 처리 이력 / 예외 처리 구조 migration
6. 회원 매칭 구조 migration
7. 로그인 / 인증 연계 범위 판단
8. 운영자 역할 부여 관리 도구
9. 비밀번호 재설정 플로우
10. 이메일 인증 활성화

아임웹 연동의 실제 다음 작업은 migration 실행이 아니라 DB 설계 문서 확정이다.

우선 `docs/PHASE_8_DB_DESIGN.md`에서
- 사이트 연결 레이어
- 상품 → membershipCode 매핑
- 포인트상품 → creditAmount 매핑
- 주문 이벤트 로그
- 회원 매칭 구조

를 먼저 고정한 뒤, 그 다음에 Supabase migration을 작성한다.

즉, 현재 단계의 우선순위는
**정책 문서 확정 → 설계 문서 확정 → migration 작성 → repository 연결 → webhook 수신 함수 추가** 순서로 본다.
