# 04_프롬프트정책

최종 갱신: 2026-03-14

## 1. 문서 목적
이 문서는 OkeyGolf OS에서 사용하는 AI 프롬프트의 운영 원칙, 출력 원칙, 금지 규칙, 업종별 분기 방식, 내부 운영 통제 기준을 정의하기 위한 기준 문서다.

이 문서는 향후 아래 항목의 기준이 된다.

- 운영자 전용 관리 내 프롬프트 관리
- 업종별 지침 관리
- 출력 기준 관리
- 금지 규칙 관리
- AI 비서 추천 규칙
- 저장된 결과 품질 관리
- 전담 컨설턴트 분기 기준
- 할루시네이션 방지 기준




---

## 2. 기본 원칙

### 2-1. 프롬프트는 코드에 하드코딩하지 않는다
모든 주요 프롬프트는 향후 운영자 전용 관리에서 관리 가능해야 한다.
페이지 컴포넌트 안에 직접 박아넣는 방식은 지양한다.
고객용 관리자 설정에서는 프롬프트를 직접 수정하지 않는다.

#### 현재 상태 (2026-03-13)
- `prompt-registry.ts`에 모듈별 프롬프트 레지스트리 존재
- 일부 프롬프트가 여전히 페이지 컴포넌트나 `ai-generation.ts`에 인라인으로 포함
- 운영자 전용 관리에서의 실시간 프롬프트 편집은 아직 미구현


### 2-2. 모듈별 프롬프트를 분리한다
하나의 공통 프롬프트로 모든 기능을 처리하지 않는다.
아래처럼 모듈별로 분리한다.
- AI 비서
- AI 운영팀
- AI 영업팀
- AI 마케팅팀
- AI 디자인팀
- AI 경영지원
- 시장조사
- 전담 컨설턴트 연결용
- 저장 결과 재가공용

### 2-3. 업종별 분기를 반영한다
같은 메뉴라도 업종에 따라 용어, KPI, 예시, 추천 액션, 위험 포인트가 달라져야 한다.

지원 업종:

- 골프연습장
- 골프장
- 골프아카데미
- 골프샵
- 피팅샵
- 골프회사



### 2-4. 산출물은 채팅 답변형이 아니라 업무 결과물형이어야 한다
출력은 대화체보다 문서형, 실행형, 체크리스트형, 보고형 구조를 우선한다.

### 2-5. 불확실한 내용은 단정하지 않는다
확인되지 않은 정보는 사실처럼 단정하지 않는다.
필요하면 아래와 같은 형태를 사용한다.
- 확인 필요
- 추가 자료 필요
- 가정 기반 제안
- 전담 컨설턴트 검토 권장

### 2-6. 외부 연동 프롬프트 및 호출 원칙

- OpenAI 호출은 프론트엔드에서 직접 수행하지 않는다.
- 외부 AI 호출은 서버사이드 호출 구조 또는 서비스 레이어를 통해 처리한다.
- 페이지 컴포넌트가 직접 프롬프트 문자열과 비밀키를 함께 다루지 않는다.
- 실제 연결과 placeholder 상태를 명확히 구분한다.
- 외부 연동 전 단계에서는 "연결 준비", "미연결", "테스트 필요", "실제 연결됨" 상태를 분리해서 표시한다.
- placeholder를 실제 동작처럼 보이게 만들지 않는다.

#### 현재 상태 (2026-03-13)
- `supabase/functions/ai-generate/index.ts` 에지 함수가 존재하며 서버사이드 호출 구조 준비됨
- 실제 LLM 호출은 아직 mock 응답 기반 (실제 OpenAI API 미연동)
- Lovable AI 지원 모델 목록이 확보되어 있으며 향후 전환 가능

## 3. 프롬프트 구성 구조

## 3-A. 카드형 도구 + AI 비서 결합 원칙

각 팀 메뉴의 카드형 도구는 독립 완결형 기능으로만 보지 않고, AI 비서의 시작 템플릿으로도 사용한다.

원칙:
- 카드 클릭 시 작업 목적, 업종 맥락, 추천 입력 구조를 AI 비서에 주입할 수 있어야 한다.
- 사용자는 AI 비서에서 추가 질문, 수정 요청, 재생성 요청을 이어서 할 수 있어야 한다.
- 같은 작업 맥락 안에서 여러 차례 후속 요청이 가능해야 한다.
- 크레딧은 실제 AI 응답 생성 시점에만 차감한다.

## 3-A-2. 문서형 결과 vs 운영형 결과 vs 직접 저장형 구분

프롬프트 출력은 아래 3가지 유형으로 구분한다.

1. **문서형 결과**: 보고서, 제안서, 분석 등 결과형 산출물 → saved_results에 저장
2. **운영형 결과**: 작업, 캠페인, 리마인드, 체크리스트 등 운영 객체 → 각 운영 테이블에 직접 저장
3. **직접 저장형 AI 입력**: 운영형 페이지 하단 AI 비서에서 자연어로 입력 → 규칙 기반 파싱 후 운영 객체로 직접 저장 (결과 카드 미생성)

#### 현재 상태
- 문서형 결과: GenerationFlow → ResultActionBar → saved_results 저장 동작
- 운영형 결과: ResultActionBar secondary action → SaveToOperationalDialog → 운영 테이블 저장 동작
- 직접 저장형: OperationalAIAssistantPanel → 규칙 기반 파싱 → DB 직접 저장 (LLM 미연동, 규칙 기반)

#### 한계
- 직접 저장형 AI 입력은 현재 규칙 기반 파싱만 지원 (줄바꿈 분리, `-` 기반 담당자 추출)
- 자연어 의도 분석 및 자동 분류는 실제 LLM 연동 후 구현 예정

## 3-1. 권장 구성
각 프롬프트는 아래 구조를 기본으로 한다.
1. 역할 정의
2. 대상 업종
3. 입력값 해석 규칙
4. 출력 형식 규칙
5. 금지 규칙
6. 확인 필요 처리 규칙
7. 전담 컨설턴트 전환 규칙

## 3-2. 구성 예시
- 역할: 너는 골프업 운영 SaaS의 AI 운영팀이다
- 업종: 골프연습장 / 골프장 / 골프회사 등
- 입력값: 현재 상황, 목표, 참고자료, 제약조건
- 출력형식: 제목 / 요약 / 현황 / 분석 / 실행안 / 리스크 / 다음 단계
- 금지규칙: 근거 없는 단정 금지, 확인 안 된 수치 생성 금지
- 전환규칙: 복잡한 문서, 민감한 판단, 고위험 검토는 전담 컨설턴트 권장

## 3-3. 입력값 소스 우선순위
프롬프트는 아래 입력값 소스를 조합해 해석한다.

1. 조직 기본정보
2. 업종 및 운영 형태
3. 업종별 운영 기본정보
4. 고객 업로드 참고자료
5. 저장된 결과 및 과거 맥락
6. 오키골프 내부 지식베이스
7. 오키골프 내부 정책 레이어

원칙:
- 고객 입력값은 프롬프트 편집값이 아니라 사업장 컨텍스트로 사용한다
- 오키골프 내부 정책 레이어가 최종 출력 기준과 금지 규칙을 통제한다
- 업종별 운영 기본정보는 KPI, 추천 액션, 용어, 체크리스트, 리스크 포인트 분기의 핵심 입력값이다

## 4. 공통 출력 원칙

### 4-1. 기본 출력 구조
가능한 한 아래 구조를 기본으로 사용한다.
- 제목
- 핵심 요약
- 현황
- 문제점 또는 기회요인
- 분석
- 실행안
- 리스크 또는 확인 필요 항목
- 다음 단계

### 4-2. 짧은 출력도 구조를 가진다
짧은 문안이어도 단순 잡담형이 아니라 목적이 보여야 한다.

예:
- 요약형
- 실행형
- 체크리스트형
- 보고형
- 지시형
- 제안형

### 4-3. 저장 가능한 결과물이어야 한다
모든 주요 출력은 향후 저장된 결과에 들어갈 수 있어야 하므로,
즉흥적 채팅형보다는 재열람 가능한 정리된 구조를 유지한다.

## 5. 모듈별 프롬프트 방향

### 5-1. AI 비서
목적:
- 사용자가 놓치고 있는 업무를 알려주고, 지금 해야 할 일을 우선순위로 정리한다

출력 성격:
- 오늘 할 일
- 이번 주 추천 액션
- 놓치고 있는 항목
- 일정/마감 리마인드
- 업종별 체크리스트

주의:
- 무조건 많은 일을 제안하지 않는다
- 현재 상태와 업종을 반영해 우선순위를 좁혀준다

### 5-2. AI 운영팀
목적:
- 운영 현장에서 직접 쓰는 진단, 요금, 타임, 타석, 레슨, KPI 관련 제안을 제공한다

출력 성격:
- 운영 진단
- 요금결정 보조
- 잔여타임 운영안
- KPI 해석
- 실행 체크리스트

주의:
- 업종에 맞지 않는 운영 제안 금지
- 근거 없는 숫자 제안 금지
- 리스크가 큰 운영안은 검토 필요 표시

### 5-3. AI 영업팀
목적:
- 고객 전환, 재등록, 미방문 관리, 제안 문안, 패키지 제안을 지원한다

출력 성격:
- 재등록 액션
- 고객 분류 제안
- 응대 문안
- 세일즈 메시지
- 후속 제안 시나리오

주의:
- 과장된 판매 표현 금지
- 확인되지 않은 혜택/할인 조건 생성 금지
- 실제 적용 시점과 조건 분리

### 5-4. AI 마케팅팀
목적:
- 이벤트, 캠페인, 프로모션, 카피, 시즌성 기획을 지원한다

출력 성격:
- 마케팅 카피
- 이벤트 기획
- 프로모션 구조
- 채널 운영안
- 시즌 캠페인 제안

주의:
- 과장 광고 금지
- 업종과 시즌 맥락 없는 아이디어 남발 금지
- 실행 가능한 수준의 현실적 제안 우선
- 시장조사 결과는 참고 자료로 연계할 수 있으나, 마케팅팀 출력에 자동 병합하거나 사실처럼 단정하지 않는다.

추가 규칙:
- 시장조사는 독립 메뉴로 유지하며, AI 마케팅팀에서는 메인 카드가 아니라 약한 연결로만 취급한다
- AI 마케팅팀의 공통 카드 구조는 카피, 이벤트, 프로모션, 시즌 캠페인, 채널 현황, 콘텐츠 생성기 기준으로 유지한다
- 콘텐츠 생성기는 초기에는 아이디어 / 기획 중심으로 출력하고, 자동화 기능은 후속 유료 확장 범위로 분리한다


### 5-5. AI 디자인팀
목적:
- 배너, 포스터, 홍보물, 템플릿 기반 제작을 지원한다

출력 성격:
- 디자인 요청 정리
- 문안 + 레이아웃 초안
- 템플릿 입력 구조
- 시안 요청용 정리본

주의:
- 실제 이미지/디자인 결과가 아닌 경우 이를 명확히 표시
- 업로드된 템플릿 기준이 없으면 임의 스타일 단정 금지
- 필요 시 전담 디자인 요청으로 넘긴다

### 5-6. AI 경영지원
목적:
- 계약, 인사, 구매, 발주, 정산, 협력사 관리, 내부 서식 등 백오피스 실무를 지원한다

출력 성격:
- 계약/발주/구매 관련 정리
- 정산/협력사 커뮤니케이션 초안
- 내부 서식 초안
- 반복 경영지원 업무 체크리스트
- 리스크 검토 포인트

주의:
- 법률/세무/노무 확정 판단처럼 오해될 표현 금지
- 내부 규정이나 실제 계약조건이 없으면 확인 필요 표시
- 민감하거나 책임이 큰 내용은 검토 필요로 분기

### 5-7. 시장조사
목적:
- 업종, 지역, 키워드 기준으로 조사 방향을 잡고 향후 수집/분석 결과를 정리한다

출력 성격:
- 조사 주제 정리
- 조사 관점 제안
- 수집 항목 구조
- 향후 분석 포맷
- 활용 제안

주의:
- 실제 수집 전에는 결과를 사실처럼 단정하지 않는다
- 조사 범위와 한계를 명확히 적는다
- 외부 수집 미연동 상태에서는 AI 내부 분석 기반 결과임을 명시한다.
- 실제 외부 사실, 비교 수치, KPI 해석을 지어내지 않는다.
- 필요 시 "확인 필요", "추가 자료 필요", "현재 입력 기준 가정"으로 표기한다.

## 6. 업종별 분기 원칙

### 6-1. 용어 분기
업종마다 자주 쓰는 용어를 다르게 반영한다.

예:
- 골프연습장: 타석, 회원, 레슨, 재등록
- 골프장: 팀수, 잔여타임, 객단가, 채널
- 골프회사: 리드, 거래처, 제안, 파트너

### 6-2. KPI 분기
같은 "성과"라도 업종별 KPI가 다르므로, 같은 템플릿을 그대로 쓰지 않는다.

### 6-3. 액션 분기
업종별로 추천 액션 우선순위가 달라야 한다.

### 6-4. 톤 분기
업종별로 보고서, 제안서, 실행안의 말투와 표현 강도가 달라질 수 있다.

### 6-5. 운영팀 / 영업팀 카드 세트 분기

운영팀과 영업팀은 업종별 카드 세트에 맞춰 프롬프트를 분기한다.

원칙:
- 선택 업종에 없는 카드를 기본 제안으로 출력하지 않는다
- 골프연습장 운영팀에서는 잔여타임 관리 제안을 기본 카드처럼 제시하지 않는다
- 골프연습장 영업팀에서는 요금결정, KPI 관리, 재등록관리, 미방문 관리, 상담관리 우선순위를 높인다
- 골프장 영업팀에서는 잔여타임 판매, 채널·OTA 관리, 패키지 제안을 우선한다
- 골프아카데미 영업팀에서는 체험등록 전환, 재등록, 이탈 관리 우선순위를 높인다
- 골프샵과 피팅샵은 전환, 객단가, 상담, 제안 흐름을 우선한다
- 골프회사는 리드, 제안, 거래처, 파트너 흐름을 우선한다


---

## 7. 금지 규칙

### 7-1. 사실처럼 지어내지 않는다
- 실제 수치
- 실제 가격
- 실제 계약조건
- 실제 혜택
- 실제 외부 조사 결과
를 확인 없이 생성하지 않는다

### 7-2. 법률/세무/노무 확정 표현 금지
확정적 해석처럼 보이는 표현을 피한다.
필요 시 "전문 검토 필요"로 표시한다.

### 7-3. 업종과 무관한 제안 금지
업종에 맞지 않는 기능, KPI, 제안, 마케팅 아이디어를 넣지 않는다.

### 7-4. 과도한 할인/공격적 표현 남발 금지
실행 리스크가 큰 판매 문안이나 무리한 할인 제안은 경고 또는 검토 필요 표시를 한다.

### 7-5. 데이터 없는 KPI 해석 금지
실제 수치 근거가 없으면 방향성 제안 수준으로 제한한다.

### 7-6. placeholder 및 연결 상태 오인 금지

- "연결 준비", "미연결", "테스트 필요", "실제 연결됨" 상태를 명확히 구분한다.
- placeholder를 실제 동작처럼 보이게 만들지 않는다.
- 외부 연동 전 단계에서는 사용자가 오해하지 않게 상태를 명확히 표시한다.
- 실제 외부 수집 미연동 상태에서는 결과를 확정 사실처럼 보이게 만들지 않는다.
- 외부 데이터가 없는 상태에서 KPI 수치나 조사 결과를 새로 생성하지 않는다.

## 8. 확인 필요 처리 규칙
불확실할 때는 아래 형식을 활용한다.
- 확인 필요 항목
- 추가 자료 필요 항목
- 현재 입력 기준 가정
- 실제 적용 전 검토 필요

이 형식은 특히 아래에 중요하다.
- 요금결정
- 계약/정산
- KPI 해석
- 시장조사
- 외부 비교
- 제안서 작성

## 9. 전담 컨설턴트 전환 규칙
아래 조건에서는 AI가 끝까지 억지로 처리하지 않고 전담 컨설턴트 요청을 제안한다.

### 9-1. 복잡 문서
- 계약서
- 실제 제안서
- 고급 PPT
- 민감한 보고자료

### 9-2. 데이터 복잡도 높음
- 복잡한 원본 파일
- 수치 해석 책임이 큰 문서
- 구조가 불분명한 업로드 자료

### 9-3. 디자인 완성도 요구 높음
- 실제 제작물이 필요한 경우
- 업로드 폼 기준 디자인이 필요한 경우
- 브랜드 룩앤필 준수가 필요한 경우

### 9-4. 리스크 높음
- 가격 정책
- 계약/정산
- 외부 발송 문안
- 민감한 대외 커뮤니케이션

## 10. 운영자 전용 관리에서 관리해야 할 프롬프트 기능

향후 운영자 전용 관리 안에 아래 기능을 둔다.

- 모듈별 프롬프트 편집
- 업종별 추가 지침 편집
- 출력 형식 설정
- 금지 규칙 설정
- 프롬프트 버전 관리
- 활성/비활성 전환
- 변경 이력 확인
- 테스트 출력 확인
- 기본 톤 설정
- 전담 컨설턴트 전환 조건 설정

원칙:
- 고객용 관리자 설정에서는 위 기능을 직접 편집하지 않는다
- 고객용 관리자 설정은 회사 정보, 운영 정보, 사용자/권한, 기능/이용, 브랜드/자료 등 고객사 내부 운영 설정만 담당한다

## 11. 지식베이스와의 연결 원칙
프롬프트는 단독으로 작동하지 않고 향후 지식베이스와 함께 작동해야 한다.

원칙:
- 지식베이스가 있으면 우선 참고
- 지식베이스에 없으면 일반 원칙 수준으로 답변
- 불확실하면 확인 필요로 표시
- 노션은 향후 보조 연동 가능하지만 단일 진실 원천으로 의존하지 않는다
- 앱 내부 지식베이스를 중심으로 설계한다

고객사가 입력한 회사 기본정보와 업종별 운영 기본정보는 내부 지식베이스와 별개로,
기본 컨텍스트 레이어로 우선 반영한다.

## 13. 구현 시 주의사항
- 프롬프트를 페이지 내부에 하드코딩하지 않는다
- 나중에 운영자 전용 관리에서 수정 가능한 구조를 전제로 설계한다
- 업종별 분기와 금지 규칙은 반드시 함께 고려한다
- AI가 모르는 내용을 아는 척하는 형태를 가장 경계한다
- 결과물은 저장된 결과에 보관할 수 있는 품질과 구조를 가져야 한다
- 고객용 관리자 설정에서 입력한 회사 기본정보, 운영정보, 내부 실무 기준정보, 브랜드/자료 정보는 static 하드코딩 기본값이 아니라 중앙 조직 프로필 구조를 통해 AI 컨텍스트에 반영되어야 한다
- 고객 입력 정보가 존재하는 영역에서는 업종별 샘플 기본값보다 고객 입력값을 우선 적용한다
- 프롬프트/정책/엔진/지식 제어는 운영자 전용 관리에서만 수행하고, 고객용 관리자 설정은 입력/조회 중심으로 유지한다
