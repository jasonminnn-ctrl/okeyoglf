# 01_메뉴구조_정본

최종 갱신: 2026-03-13

## 1. 문서 목적
이 문서는 OkeyGolf OS의 최종 대메뉴 구조와 각 메뉴의 역할을 고정하기 위한 기준 문서다.
향후 러버블 구현, 화면 설계, 저장 구조, 권한 구조, 관리자 설정, 프롬프트 구성은 이 메뉴 구조를 기준으로 확장한다.
메뉴는 임의로 흔들지 않고, 변경이 필요할 경우 반드시 이 문서를 기준으로 검토한다.

## 2. 메뉴 설계 원칙
- 대메뉴는 사용자가 바로 이해할 수 있는 실무형 이름을 사용한다
- 업종별로 메뉴를 바깥에서 나누지 않고, 공통 메뉴 안에서 내부적으로 분기한다
- AI 기능은 각 메뉴 내부에서 작동하며, 사용자에게는 "팀" 또는 "업무" 단위로 보이게 한다
- ROS 엔진명은 사용자 대메뉴로 직접 노출하지 않는다
- AI로 처리 가능한 일과 전담 컨설턴트 요청이 자연스럽게 연결되어야 한다
- 모든 주요 메뉴에는 전담 컨설턴트 요청 CTA를 둘 수 있어야 한다
- 저장된 결과와 관리자 설정은 전체 시스템의 공통 기반 메뉴로 유지한다

## 4. 대메뉴별 정의

## 4-A. 팀별 AI 작업 방식 원칙

각 팀 메뉴는 아래 구조를 기본으로 한다.

1. 상단 도구 카드 영역
2. 하단 AI 비서 영역

원칙:
- 팀별 기능 도구는 구조화된 입력과 빠른 결과 생성을 위한 도구형 플로우로 유지한다.
- 하단 AI 비서는 각 팀 맥락에 맞는 자유형 요청/수정/추가 공간으로 운영한다.
- 고객 화면에서 "AI 작업실", "보조 AI 작업실" 표현은 사용하지 않는다. 모두 "AI 비서"로 통일한다.
- `전담 컨설턴트 요청`은 AI 비서의 대체가 아니라, 사람/운영 인계 목적의 별도 액션으로 유지한다.
- 결과 생성 후 `전담 컨설턴트 요청` 버튼은 향후 실제 요청 상태(`대기`, `접수`, `처리중`, `완료`)로 이어지는 운영 흐름으로 확장한다.

## 4-B. 팀 페이지 작업 구조 원칙

AI 비서 / AI 운영팀 / AI 영업팀 / AI 마케팅팀 / AI 디자인팀 / AI 경영지원 / 시장조사 페이지는 아래 2단 구조를 기본으로 한다.

1. 상단 도구 카드 영역
2. 하단 AI 비서 영역

상단 도구 카드는 2종으로 구분한다:
1) 운영형 도구 카드 — 즉시 기능 페이지로 진입. 반복 실행, 상태관리, 담당자, 기한, 체크, 추적이 필요한 카드.
2) 보조형 카드 — AI 비서에 프롬프트/컨텍스트를 넘기는 카드. 요약형, 실행 시작점 역할.

## 4-B-2. 운영형 페이지 공통 패턴

운영형 페이지(운영 점검, 캠페인, 리마인드, 체크리스트)는 아래 하이브리드 구조를 기본으로 한다.

[상단] 직접 입력/수정 가능한 실제 툴 UI
- 리스트/보드/카드형 운영 객체
- 내보내기 액션 (CSV/XLSX)
- 상태/담당자/완료자/완료일 표시

[하단] AI 비서
- 자연어로 추가/수정/분배/초안 요청 가능
- 처리 결과 중심 (장문 채팅 응답 금지)
- 최근 처리 결과 1~3개만 짧게 노출

원칙:
- 툴만 있는 페이지 금지
- 채팅만 있는 페이지 금지
- 수동입력 + AI입력 + 추적 + export 하이브리드 필수
- 고객 화면에서 "AI 작업실" 표현 제거, 모두 "AI 비서"로 통일

### 현재 구현 상태 (2026-03-13)
- 4종 운영형 페이지 모두 하이브리드 패턴 적용 완료
- OperationalAIAssistantPanel 공통 컴포넌트 적용
- OperationalExportMenu (CSV/XLSX) 공통 적용
- OperationalMetaBadges (담당자/완료자/완료일 표시) 공통 적용
- AI 비서 하단 패널은 현재 규칙 기반 파싱 (실제 LLM 호출 미연동)

AI 비서 내부에서 아래 카드는 운영형 도구로 고정한다:
- 놓치고 있는 운영 항목 → `/ai-assistant/ops-check`
- 캠페인 추천 → `/ai-assistant/campaign-planner`
- 일정/마감 리마인드 → `/ai-assistant/reminder-board`
- 업종별 체크리스트 → `/ai-assistant/checklist-manager`

오늘의 할 일 / 이번 주 추천 액션은 보조형 카드(AI 비서 프롬프트 주입)로 유지.

원칙:
- 운영형 카드 클릭 시 해당 기능 페이지로 직접 이동한다.
- 보조형 카드 클릭 시 하단 AI 비서에 해당 작업용 프롬프트와 컨텍스트를 주입할 수 있어야 한다.
- 전담 컨설턴트 요청은 AI 비서의 대체가 아니라 사람/운영 인계용 별도 액션으로 유지한다.

### 4-1. 대시보드
역할:
- 서비스 진입 메인 화면
- 현재 상태를 한눈에 확인하는 운영 관제 센터

포함 요소:
- 핵심 KPI
- 최근 생성 결과
- 최근 시장조사
- 최근 전담 컨설턴트 요청
- 이번 주 추천 액션
- 현재 선택 업종
- 멤버십 등급
- 향후 크레딧 요약 영역

### 4-2. AI 비서
역할:
- 사용자가 놓치고 있는 운영 업무를 리마인드하고 다음 액션을 제안하는 실행 비서

포함 요소:
- 오늘의 할 일
- 이번 주 추천 액션
- 놓치고 있는 운영 항목
- 캠페인 추천
- 일정/마감 리마인드
- 업종별 체크리스트
- 빠른 전담 요청 CTA

메인 랜딩 하단 영역:
- AI 비서 메인 랜딩 하단은 자유형 AI 입력창이 아니라 **공지사항** 영역으로 운영한다.
- 공지사항 영역은 운영 공지, 서비스 업데이트, 이벤트, 업종별 운영 팁, 중요 점검 안내를 카드 리스트 형태로 보여준다.
- 각 운영형 메뉴 내부 페이지(운영 점검/캠페인/리마인드/체크리스트)는 `직접 입력 + AI 비서 + 결과 관리` 패턴을 유지한다.
- 공지사항은 향후 `/operator` 운영자 관리 화면에서 등록한 공지가 내려오는 구조로 확장 예정.
- 현재 상태: mock 데이터 기반 (공지 관리 기능 연동 예정)

### 4-3. AI 운영팀

역할:
- 실제 운영 현장에서 자주 쓰는 운영 실무 기능 제공
- 단, 운영팀 내부 카드는 업종에 따라 다르게 분기한다

원칙:
- 대메뉴명은 공통 유지
- 내부 카드는 업종별로 다르게 노출
- 운영팀은 운영 실무 / 배정 / 시설 / 현장 흐름 중심으로 유지
- 가격정책, KPI, 재등록 전환 같은 매출 직접 연관 항목은 업종에 따라 영업팀으로 이동 가능

운영 지원 전담 컨설턴트 CTA


### 4-4. AI 영업팀

역할:
- 고객관리, 전환, 재등록, 상담, 가격정책, 매출과 직접 연결되는 기능 제공
- 단, 영업팀 내부 카드는 업종에 따라 다르게 분기한다

원칙:
- 업종별 고객 / 리드 / 거래처 구조에 맞춰 카드가 달라진다
- 영업팀은 전환, 유지, 가격, 상담, KPI 중심으로 유지한다

영업 지원 전담 컨설턴트 CTA

역할:
- 캠페인, 카피, 프로모션, 콘텐츠 기획 등 마케팅 실행 지원

포함 요소:
- 마케팅 카피 생성기
- 이벤트 생성기
- 프로모션 기획
- 시즌 캠페인 제안
- 채널 현황
- 콘텐츠 생성기
- 마케팅 지원 전담 컨설턴트 CTA

원칙:
- 시장조사는 독립 대메뉴로 유지한다
- AI 마케팅팀에는 시장조사 메인 카드를 두지 않는다
- 필요 시 결과 화면 또는 하단 CTA에서만 약하게 연결한다

### 4-8. 시장조사
역할:
- 지역, 업종, 키워드 중심의 조사 요청 및 향후 수집/분석 확장 기능

포함 요소:
- 업종
- 지역
- 키워드
- 수집 개수
- 조사 시작 버튼
- 최근 작업
- 결과 영역
- 전담 컨설턴트 연결 CTA

#### 운영 원칙

- 시장조사는 조사 요청 생성, 최근 작업 확인, 저장된 결과 연결까지 포함하는 **독립 워크플로우 메뉴**다.
- 시장조사 결과는 저장된 결과와 연결되며, 필요 시 전담 컨설턴트로 전환된다.
- 외부 수집 엔진 미연동 상태에서는 **AI 내부 분석 기반 결과**임을 구분해서 표시한다.
- 실제 외부 조사 결과, 비교 수치, KPI 해석을 지어내지 않는다.
- 필요 시 "확인 필요", "추가 자료 필요", "현재 입력 기준 가정"으로 표기한다.
- 시장조사는 AI 마케팅팀과 연결될 수 있으나, 마케팅팀에 종속되지 않는다.

### 4-11. 관리자 설정
역할:
- 고객사가 자신의 회사 정보, 운영 정보, 내부 사용자 및 기본 사용 환경을 설정하는 고객용 설정 메뉴
- 서비스가 업종별/조직별 컨텍스트를 정확히 이해하도록 사업장 기본값과 내부 운영 기준을 유지하는 입력 화면
- 현재 이용 상태를 고객이 직접 확인할 수 있는 조회 화면을 함께 제공하는 메뉴

포함 요소:
- 회사 기본정보
- 업종 및 운영 형태 설정
- 업종별 운영 기본정보
- 내부 실무 기준정보
- 사용자 관리
- 사용자별 권한 설정
- 기능 및 이용 설정
- 브랜드/자료 설정
- 현재 멤버십 조회
- 업그레이드 안내
- 현재 크레딧 잔액 조회
- 최근 크레딧 사용내역 조회
- 현재 활성 기능/제한 기능 요약 조회


화면 구성 원칙:
- 고객용 관리자 설정은 상단 탭 또는 sticky 서브탭 구조로 정리한다
- 탭은 아래 5개로 사용한다
  - 회사/업종
  - 운영정보
  - 사용자/권한
  - 기능/브랜드
  - 이용현황
- 세로 스크롤이 과도하게 길어지지 않도록 각 탭 안은 카드형 섹션으로 정리한다

권한 분리 원칙:
- 고객용 관리자 설정은 고객사 내부 운영에 필요한 설정과 조회 기능만 다룬다
- 고객용 관리자 설정에서는 프롬프트, 정책, 엔진, 지식, ROS, 멤버십 정책, 크레딧 정책, 기능 노출/비노출 정책을 직접 수정하지 않는다
- 멤버십과 크레딧은 조회 전용으로만 노출하고 정책 편집은 운영자 전용 관리에서만 수행한다

제외 원칙:
- 프롬프트 관리
- 업종별 지침 관리
- 출력 기준 관리
- 금지 규칙 관리
- 지식베이스 관리
- 참고자료/출처 관리
- AI 참조 허용 범위 설정
- 엔진 설정
- ROS 정책
- 멤버십 정책
- 크레딧 정책
- 기능 노출/비노출 설정
- 전담 컨설턴트 전환 규칙
위 항목은 고객용 관리자 설정에 두지 않고 운영자 전용 관리에서만 다룬다

### 5-3. 운영자 외부 연동 탭 (고객 비노출)

- 운영자 전용 `/operator` 내부에 `외부 연동` 탭을 둔다.
- 이 탭은 고객용 관리자 설정 5탭과 분리한다.
- 외부 연동 탭에서는 아래 내용을 다룬다.
  - 연동 상태
  - OpenAI 연동
  - 아임웹 회원/주문 연동
  - 상품 → 멤버십 매핑
  - 포인트상품 → 크레딧 매핑
  - 알림/발송 연동
  - 동기화 이력 / 예외 처리
- 고객용 사이드바 11개 구조는 변경하지 않는다.
- 외부 연동 관련 세부 운영은 운영자 화면에서만 관리한다.

## 8. 메뉴 연결 원칙
- AI 비서는 전체 메뉴를 가로지르는 추천 허브 역할을 한다
- 각 팀 메뉴에서 생성된 결과는 저장된 결과로 연결된다
- 각 팀 메뉴에는 전담 컨설턴트 요청 CTA를 둘 수 있다
- 고객용 관리자 설정은 회사 기본정보와 운영정보를 입력하는 역할을 한다
- 운영자 전용 관리는 모든 모듈의 출력 품질, 정책, 지식 사용 방식, 멤버십, 크레딧, 기능 노출/비노출을 통제한다
- 시장조사는 마케팅팀, 운영팀, 전담 컨설턴트와 모두 연결 가능해야 한다
- 고객용 사이드바에는 기존 11개 대메뉴만 노출한다
- 운영자 관리와 엔터프라이즈는 고객용 대메뉴에 포함하지 않고 내부 role 기반 숨김 route로 운영한다
- 고객용 관리자 설정에서 입력한 회사/운영/브랜드 정보는 중앙 조직 프로필 구조를 통해 AI 생성 컨텍스트에 반영되어야 한다
- 고객용 관리자 설정의 이용현황은 조회 전용으로만 노출하고, 멤버십/크레딧/기능 정책의 직접 편집은 운영자 전용 관리에서만 수행한다


## 9. 구현 시 주의사항
- 메뉴명을 개발 중 임의로 다시 추상화하지 않는다
- 기존 생성기 페이지는 삭제하지 않고 새 메뉴 구조 아래 재배치한다
- ROS 엔진을 사용자 메뉴로 직접 노출하지 않는다
- 상담팀은 독립 대메뉴로 두지 않는다
- 마케팅팀과 디자인팀은 분리 유지한다
- 경영지원실이 아니라 경영지원 명칭을 사용한다
